# VoLTE Mutation Fuzzer — 발전 로드맵

> [!abstract] 요약
> Phase 1~4(프로토콜 모델 → Generator → Mutator → Sender)는 구현 완료. 프로젝트의 다음 과제는 **"패킷을 보낼 수 있다"에서 "캠페인을 돌리고 결함을 탐지한다"로의 전환**이다. 핵심 미해결 축은 Oracle(이상 탐지), Controller(캠페인 자동화), 실험 환경 확정이다.

---

## 1. 현재 위치 진단

### 1.1 Phase 완료 현황

| Phase | 내용 | 상태 | 비고 |
|-------|------|:----:|------|
| Phase 1 | SIP 메시지 분류 + 카탈로그 | ✅ | 14 메서드 + 75 응답코드, IANA 전수조사 완료 |
| Phase 2 | Generator (베이스라인 생성) | ✅ | 전체 메서드/응답 지원, CLI 통합 |
| Phase 3 | Mutator (3계층 뮤테이션) | ✅ | model/wire/byte, 시드 기반 재현 |
| Phase 4 | Sender/Reactor (전송/수집) | ✅ | softphone + real-ue-direct 구현 |
| Phase 5 | Oracle (이상 탐지) | ❌ | 미착수 |
| Phase 6 | Controller (캠페인 오케스트레이션) | ❌ | 미착수 |

### 1.2 현재 할 수 있는 것

```bash
# 단일 패킷 생성 → 뮤테이션 → 전송 파이프라인
uv run fuzzer request INVITE \
  | uv run fuzzer mutate packet --strategy state_breaker --seed 42 \
  | uv run fuzzer send packet --target-host 10.20.20.2 --target-port 5060
```

- 14개 메서드, 75개 응답코드에 대한 정상 패킷 생성
- model/wire/byte 3계층 뮤테이션 적용
- softphone 또는 실 UE(direct)에 단일 패킷 전송
- 소켓 응답 수집 및 outcome 분류 (provisional/success/error/timeout)

### 1.3 현재 할 수 **없는** 것

| 영역 | 부재 내용 |
|------|----------|
| **이상 탐지** | 크래시, ANR, 재부팅, SIP 스택 비정상 자동 감지 없음 |
| **캠페인 자동화** | 다수 테스트 케이스 순차/병렬 실행 불가 |
| **결과 집계** | 실행 결과 DB/리포트 없음 |
| **재현 자동화** | 시드 기반 재현은 가능하나 자동 replay 루프 없음 |
| **P-CSCF 경유** | real-ue-pcscf 인터페이스만 존재, 구현 없음 |
| **관측 플러그인** | siptrace, pcap, ADB/logcat 미연동 |

### 1.4 강점과 약점

> [!success] 강점
> - SIP 표면 인벤토리가 **전수 수준**으로 깊다
> - Generator → Mutator → Sender 책임 경계가 명확
> - 문서, 카탈로그, 구현이 **같은 축**을 보고 있다
> - "단말 지향 퍼징"이라는 문제 정의가 흔들리지 않는다
> - 시드 기반 결정적 재현 가능

> [!warning] 약점
> - **실행 계층**(Oracle/Controller)이 완전히 비어 있다
> - 공격면 우선순위 문서는 있으나 **실행 계획으로 접히지 않았다**
> - 관측이 소켓 응답에만 의존 — real-ue에서는 부족
> - 대상 단말(ISSUE-01)과 관측 채널(ISSUE-05) **미결정**
> - 소프트폰 실제 검증 미완

## 2. 미해결 의사결정

다음 단계로 넘어가기 전에 **반드시 결정해야 하는** 핵심 질문들이다.

### 2.1 ISSUE-01: 초기 타깃 단말 선정

| 선택지 | 장점 | 단점 |
|--------|------|------|
| **Android (특정 모델)** | ADB/logcat 접근 용이, 크래시 로그 풍부 | 제조사별 SIP 스택 차이 |
| **iOS** | VoLTE 사용률 높음 | 관측 채널 극히 제한 |
| **소프트폰 (Baresip/PJSIP)** | 완전 제어 가능, 로그 풍부 | 실 단말 대표성 부족 |

> [!question] 결정 필요
> 1차 실험 대상은 **소프트폰 sanity** → **Android 실 단말** 순서가 현실적.
> 구체적 모델·OS 버전은 실험 환경 확보 후 확정.

### 2.2 ISSUE-05: Oracle 관측 채널 선택

| 채널 | 감지 가능 | 구현 난이도 |
|------|----------|:----------:|
| **소켓 응답 분석** | timeout, 비정상 응답 코드, 응답 부재 | 낮음 (이미 구현) |
| **ADB logcat** | 크래시, ANR, 프로세스 재시작 | 중간 |
| **ADB bugreport** | 시스템 레벨 크래시 덤프 | 중간 |
| **네트워크 캡처 (tcpdump/pcap)** | 패킷 도달 여부, 중간 드롭 | 중간 |
| **디바이스 상태 폴링** | 재부팅, 화면 상태, 프로세스 생존 | 낮음 |

> [!question] 결정 필요
> 최소 구현은 **소켓 응답 + ADB logcat + 디바이스 상태 폴링** 조합이 적절.

### 2.3 기타 미결 사항

| 항목 | 현재 상태 | 필요한 결정 |
|------|----------|-----------|
| IMS 인증(AKA) 시뮬레이션 | 미구현 | REGISTER/인증 완료 상태를 전제할 것인가 |
| real-ue-pcscf 구현 시점 | 인터페이스만 존재 | direct 경로 검증 후 진행 여부 |
| TCP/TLS 전송 우선순위 | UDP 우선 결정 완료 | TCP/TLS는 언제 추가할 것인가 |
| 소프트폰 선택 | Baresip 런처 구현 | Linphone/PJSIP 통합 시점 |

## 3. 단기 로드맵 (Phase 5~6)

### Phase 5: Oracle — 이상 탐지 엔진

> [!info] 목표
> "패킷을 보낸 뒤 단말에 무슨 일이 일어났는가"를 **자동으로 판정**하는 계층.

#### 5.1 아키텍처

```
                    ┌──────────────────────┐
                    │    Oracle Engine      │
                    │                      │
 SendReceiveResult ─┤  ┌── Socket Observer │
                    │  ├── ADB Observer    │──→ OracleVerdict
 (from Sender)     │  ├── Device Poller   │    {normal|suspicious|
                    │  └── Pcap Observer   │     crash|anr|reboot|
                    │                      │     timeout|unknown}
                    └──────────────────────┘
```

#### 5.2 핵심 구현 항목

| 컴포넌트 | 역할 | 우선순위 |
|---------|------|:--------:|
| `OracleVerdict` | 판정 결과 데이터 모델 | P0 |
| `SocketOracle` | 소켓 응답 기반 판정 (이미 부분 구현) | P0 |
| `ADBObserver` | logcat 스트림 실시간 감시 | P1 |
| `DevicePoller` | 주기적 디바이스 상태 확인 (ping, process check) | P1 |
| `PcapObserver` | 네트워크 캡처 기반 패킷 도달 확인 | P2 |
| `OracleAggregator` | 다중 관측 결과 종합 판정 | P1 |

#### 5.3 판정 기준 설계

```
OracleVerdict:
  NORMAL          — 정상 SIP 응답, 디바이스 정상
  SUSPICIOUS      — 비정상 응답 패턴 (예: 예상과 다른 코드, 지연)
  TIMEOUT         — 응답 없음 + 디바이스는 생존
  CRASH           — logcat에서 SIP 프로세스 crash 감지
  ANR             — logcat에서 ANR 패턴 감지
  REBOOT          — 디바이스 연결 끊김 후 재등장
  STACK_FAILURE   — SIP 스택 비정상 (등록 해제, 통화 불가)
  UNKNOWN         — 판정 불가
```

#### 5.4 ADB Observer 상세

```python
# 감시 대상 logcat 패턴
CRASH_PATTERNS = [
    r"FATAL EXCEPTION",
    r"Native crash",
    r"Tombstone written",
    r"Process .* has died",
]

ANR_PATTERNS = [
    r"ANR in",
    r"Input dispatching timed out",
    r"Reason: executing service",
]

SIP_STACK_PATTERNS = [
    r"oImsRegistration.*DEREGISTERED",
    r"oImsCallSession.*ERROR",
    r"oTelephony.*LOST_SERVICE",
]
```

> [!tip] 점진적 구현
> 1단계: Socket Oracle만으로 softphone 실험
> 2단계: ADB Observer 추가하여 Android 실 단말 실험
> 3단계: 복합 판정 (Aggregator)

### Phase 6: Controller — 캠페인 오케스트레이션

> [!info] 목표
> 다수의 뮤테이션 케이스를 **자동 실행**, **결과 기록**, **재현 지원**하는 캠페인 관리 계층.

#### 6.1 아키텍처

```
┌──────────────────────────────────────────────────────┐
│                   Campaign Controller                 │
│                                                      │
│  CampaignConfig ──→ CaseGenerator ──→ Executor       │
│  (target, scope,    (method/code     (send + wait    │
│   strategy, seed     selection,       + observe)     │
│   concurrency)       mutation plan)                  │
│                                        │             │
│                                    OracleEngine      │
│                                        │             │
│                                   ResultStore        │
│                                   (JSON/SQLite)      │
│                                        │             │
│                                   ReportGenerator    │
└──────────────────────────────────────────────────────┘
```

#### 6.2 핵심 구현 항목

| 컴포넌트 | 역할 | 우선순위 |
|---------|------|:--------:|
| `CampaignConfig` | 캠페인 설정 (대상, 범위, 전략, 시드) | P0 |
| `CaseGenerator` | 공격면 Tier 기반 테스트 케이스 자동 생성 | P0 |
| `CampaignExecutor` | 순차/병렬 케이스 실행 루프 | P0 |
| `ResultStore` | 실행 결과 영속 저장 (JSON → SQLite) | P0 |
| `CampaignReporter` | 요약 리포트 생성 (통계, 이상 목록) | P1 |
| `ReplayEngine` | 시드 기반 특정 케이스 재현 | P1 |
| `CampaignScheduler` | Tier 우선순위 기반 케이스 순서 결정 | P2 |

#### 6.3 CLI 설계

```bash
# 캠페인 실행
uv run fuzzer campaign run \
  --target-host 10.20.20.2 \
  --target-port 5060 \
  --scope tier1 \
  --strategy state_breaker \
  --max-cases 1000 \
  --timeout 5 \
  --output results/campaign-001.json

# 캠페인 결과 조회
uv run fuzzer campaign report results/campaign-001.json

# 특정 케이스 재현
uv run fuzzer campaign replay results/campaign-001.json --case-id 42

# 이상 케이스만 필터
uv run fuzzer campaign report results/campaign-001.json --filter crash,anr,reboot
```

#### 6.4 캠페인 실행 흐름

```
for each case in CaseGenerator(config):
    1. Generate baseline packet (method/code from scope)
    2. Apply mutation (strategy + layer + seed)
    3. Send to target (Sender)
    4. Observe response (Oracle)
    5. Record result (ResultStore)
    6. If verdict == CRASH|ANR|REBOOT:
         → mark as finding
         → log full reproduction info (seed, method, strategy)
    7. If verdict == REBOOT:
         → wait for device recovery
         → verify device state before next case
    8. Cooldown between cases (configurable)
```

#### 6.5 결과 저장 스키마

```json
{
  "campaign_id": "campaign-001",
  "started_at": "2026-03-24T10:00:00Z",
  "config": { "scope": "tier1", "strategy": "state_breaker", ... },

    {
      "case_id": 0,

    {

      "finding": {
        "logcat_snippet": "FATAL EXCEPTION in com.oem.ims...",
```

## 4. 중기 로드맵

Phase 5~6 안정화 이후 진행할 확장 영역.

### 4.1 P-CSCF 경유 경로 (`real-ue-pcscf`)

| 항목 | 내용 |
|------|------|
| **목적** | IMS 네트워크 경유 전송으로 Tier 3 공격면(IMS private headers) 실험 |
| **전제** | Open5GS + Kamailio 환경에서 P-CSCF 라우팅 검증 완료 |
| **구현** | `SIPSenderReactor`에 pcscf 모드 추가, P-CSCF 주소 해석 |
| **가치** | `P-Asserted-Identity`, `P-Charging-*`, `Feature-Caps` 등 IMS 전용 표면 자극 |

### 4.2 다중 관측자 통합

```
Observer Plugin Architecture:
  ┌─────────────┐
  │ ObserverBase │  ← abstract
  ├─────────────┤
  │ SocketObs   │  ← SIP 응답 분석
  │ ADBObs      │  ← logcat/bugreport
  │ PcapObs     │  ← tcpdump/tshark
  │ DevicePoll  │  ← 생존/상태 확인
  │ SIPTraceObs │  ← Kamailio siptrace
  └─────────────┘
```

- 각 Observer는 독립 실행, 결과를 `OracleAggregator`에 전달
- 플러그인 구조로 환경별 Observer 조합 가능

### 4.3 IMS 인증 시뮬레이션

| 단계 | 내용 |
|------|------|
| 1단계 | REGISTER 완료 상태 가정 (현재) |
| 2단계 | Digest 인증 challenge-response 자동 처리 |
| 3단계 | IMS AKA (3GPP TS 33.203) 시뮬레이션 |

- Generator에 인증 컨텍스트 주입 기능 추가
- Mutator에 인증 필드 특화 뮤테이션 전략 추가

### 4.4 TCP/TLS 전송 지원

| 항목 | 현재 | 목표 |
|------|------|------|
| UDP | ✅ 구현 완료 | 유지 |
| TCP | ✅ Sender에 구현 | 캠페인 통합 |
| TLS | ❌ 미구현 | SIPS URI 지원 + TLS 소켓 |
| WebSocket | ❌ | RFC 7118 기반 WS/WSS |

### 4.5 소프트폰 확장

| 소프트폰 | 용도 | 통합 방식 |
|---------|------|----------|
| **Baresip** | CLI 자동화 | ✅ 런처 구현, 설정 기반 |
| **Linphone** | GUI 수동 검증 | 수동 실행, 로그 수집 |
| **PJSIP/PJSUA2** | 프로그래밍 자동화 | Python 바인딩, 콜백 기반 |

> [!tip] 추천 순서
> Baresip (현재) → PJSIP (자동화 강화) → Linphone (수동 보조)

## 5. 장기 로드맵

### 5.1 스마트 뮤테이션 (ML/Feedback-Driven)

| 접근 | 설명 |
|------|------|
| **Coverage-Guided** | 코드 커버리지 피드백으로 뮤테이션 방향 조정 (requires instrumentation) |
| **Grammar-Aware** | SIP 문법 트리 기반 구조적 뮤테이션 생성 |
| **Reinforcement Learning** | 과거 crash 발견 패턴에서 다음 뮤테이션 전략 학습 |
| **Corpus Distillation** | 유효한 뮤테이션 집합을 축소하여 효율 극대화 |

> [!note] 현실적 접근
> 블랙박스 퍼징이므로 coverage-guided는 어렵다. **Grammar-aware + corpus distillation**이 현실적 첫 단계.

### 5.2 병렬/분산 퍼징

```
Campaign Controller
  ├── Worker 1 (Android UE #1, Tier 1 mutations)
  ├── Worker 2 (Android UE #2, Tier 2 mutations)
  ├── Worker 3 (Softphone, Tier 4 byte mutations)
  └── Aggregator (결과 통합, 중복 제거)
```

- 다수 단말에 동시 퍼징
- Worker별 독립 시드 공간
- 중앙 결과 집계 + 중복 finding 제거

### 5.3 크로스 플랫폼 단말 지원

| 플랫폼 | 관측 방법 | 난이도 |
|--------|----------|:------:|
| Android | ADB logcat + bugreport | 중간 |
| iOS | 제한적 (sysdiagnose, CrashReporter) | 높음 |
| 임베디드 IoT | UART/시리얼 로그 | 높음 |
| 자동차 IVI | CAN + 시리얼 | 매우 높음 |

### 5.4 퍼징 결과 분석 대시보드

```
┌─────────────────────────────────────────────┐
│             Fuzzing Dashboard               │
├─────────────┬───────────────┬───────────────┤
│ Campaign    │  Finding      │  Coverage     │
│ Progress    │  Timeline     │  Heatmap      │
│ (실시간)     │  (크래시 발견) │  (메서드별)    │
├─────────────┴───────────────┴───────────────┤
│ Method × Strategy × Layer Matrix            │
│ (어떤 조합이 가장 많은 이상을 발견했는가)       │
├─────────────────────────────────────────────┤
│ Top Findings (재현 명령어 포함)               │
└─────────────────────────────────────────────┘
```

### 5.5 CI/CD 통합 (Regression Fuzzing)

```yaml
# .github/workflows/fuzz-regression.yml
on:
  schedule:
    - cron: '0 2 * * *'  # 매일 새벽 2시

jobs:
  fuzz:
    runs-on: self-hosted  # UE 연결된 러너
    steps:
      - run: uv run fuzzer campaign run --scope tier1 --max-cases 500
      - run: uv run fuzzer campaign report --fail-on crash,anr,reboot
```

## 8. 기술 부채 및 개선 사항

현재 코드베이스에서 로드맵 진행 시 함께 해결해야 할 항목.

| 영역 | 내용 | 관련 Phase |
|------|------|-----------|
| **MSISDN 해석 경직성** | Docker 컨테이너 이름 하드코딩 (`VMF_REAL_UE_SCSCF_CONTAINER`) | Phase 5~6 |
| **비동기 미지원** | 동기 소켓 I/O만 사용, 다수 대상 병렬 전송 시 병목 | Phase 6+ |
| **Observer 구조** | 설계됐으나 미구현, 플러그인 인터페이스 확정 필요 | Phase 5 |
| **에러 핸들링** | Sender에서 네트워크 오류 시 graceful recovery 부족 | Phase 6 |
| **설정 관리** | 환경변수 기반 — 캠페인 설정은 파일 기반이 더 적절 | Phase 6 |
| **로깅** | 구조화된 로깅 미도입 (structlog 등) | Phase 5~6 |
