오늘은 **wire 레이어 fuzz strategy 3개 추가 → bypass 모드 응답 수신 fix → oracle FP 재정의 → 픽셀 attach 디버깅** 순서로 진행했다.

## 1. wire 레이어 새 strategy 3개

기존 `--strategy default` 가 `mutate_header_value` 안의 `null_byte` / `overflow` / `garbage` / `wrong_format` / `crlf_inject` / `empty` 6 개 중 1/6 확률로만 골라서 fuzz 다양성이 약했다. 그 위에 8 개의 `_mutate_wire_*` operator 분배까지 거치니 통계상 **240 케이스 중 null byte 변이 1건 정도** 밖에 안 들어갔다 (오늘 아침 3개 캠페인 직접 확인).

그래서 **목적별로 강제 변이**하는 wire strategy 3개를 추가:

- **`null_byte_only`** — 매 케이스 random 헤더 1개의 값에 null byte 끼움. variant: `prefix` / `midpoint` / `suffix` / `scrub`.
- **`boundary_only`** — 헤더 값을 boundary 정수/길이/특수값으로 통째 교체. variant: `numeric` / `negative_overflow` / `zero_length` / `single_char` / `huge_length`.
- **`byte_edit_only`** — 헤더 값에 1byte 단위 surgical edit. variant: `trim_last/first/random`, `flip_last/first/random`, `dup_byte`, `swap_adjacent`, `insert_byte`, `insert_null`.

각 case 의 jsonl record 의 mutation_record 의 `note` 필드에 `variant=...` 가 적혀 결과 분석 시 분리 가능.

### 1.1 silent 0-case 버그

`null_byte_only` 처음 추가하고 캠페인 돌렸더니 `total: 0 / status: completed` 로 즉시 종료. 에러도 안 나옴.

원인: `campaign/core.py:91` 의 `_SUPPORTED_STRATEGIES` 가 `mutator/profile_catalog.py:SUPPORTED_STRATEGIES_BY_LAYER` 의 **하드코드 복사본**. 주석에까지 “mutator/core.py 와 동기화” 라고 쓰여있었는데 새 strategy 추가 시 한 쪽만 고쳐서 generator 가 spec 0개 만들고 정상 종료한 것.

silent 한 게 가장 나쁘다. 등록 누락 시 ValueError 나도록 강화하든가 lookup 을 단일 출처로 통일하는 게 다음 청소 대상.

### 1.2 EMSGSIZE

`boundary_only` 의 `huge_length` variant 에 `1048576` (1MB) 길이 항목이 있어서 헤더 1개에 `A` × 1MB 가 들어가버려 UDP send 에서 `OSError: [Errno 90] Message too long`. UDP 한 datagram 한계 65507 초과면 atomic send 불가. boundary 의미는 살리되 의미 있는 범위로 잘라 `(1, 256, 1024, 4096, 8192)` 로 변경.

## 2. bypass 모드 응답 수신 fix

오전부터 사용자가 “bypass 캠페인은 다 timeout 인데 native 는 다 받음” 이라고 보고. 같은 seed range 의 jsonl 비교했을 때 bypass 의 case 0~4 모두 timeout, raw_response 비어있음.

처음 잡은 가설은 **kamailio 가 P-CSCF netns 의 5060 wildcard 로 응답을 가로챈다** 였고 — 그건 맞았다. 다만 fix 방향을 처음엔 잘못 잡았다.

### 2.1 처음 시도 (잘못된 fix)

`_send_via_container` 에서 `bind_port == 0` (ephemeral) 일 때 P-CSCF 의 xfrm port_map 으로 6107 (port_ps) 을 resolve 해서 그 값으로 bind, Via rewrite, connect peer 까지 일치시키려 함. 동작상으론 native 와 비슷한 모양.

근데 사용자가 “bypass 는 P-CSCF 안 거치고 UPF 로 보내려는 거야” 그리고 “**15100 으로 응답 올걸**” 이라고 정정. `mt_local_port` 가 `15100` 인 점, `sender/cli.py:357-371` 의 single-shot 분기에서 이미 `null/bypass` 시 `bind_port = mt_local_port` 로 설정하고 있던 점을 확인하니 일관된 그림은 별 거 아니었다 — **campaign 모드만 그 dispatch 를 안 거쳐서 bind_port 가 빠져있던 누락 버그**.

### 2.2 진짜 fix

```python
# campaign/core.py:_target 생성
self._target = TargetEndpoint(
    ...
    bind_container=config.bind_container,
    bind_port=(
        config.mt_local_port
        if config.ipsec_mode in ("null", "bypass")
        else None
    ),
)
```

이거 한 줄로 sender single-shot 과 동일한 의도 (bypass = 5100 port-bind / Via sent-by 일치 / kamailio 우회) 가 campaign 에도 적용. 잘못 추가했던 xfrm port_map fallback 은 revert.

부수적으로 `container_exec.py` 의 UDP driver 에 `sock.connect((remote_host, remote_port))` 추가해서 kamailio 의 wildcard listener 보다 4-tuple match 가 우선되게 했다. native driver 의 패턴을 그대로 따른 거라 소화 부담 적다.

## 3. RILJ Unexpected response — 진짜 critical 만 잡게

`oem_ril_error` 패턴이 case-insensitive 라 `RILJ ... processResponse: Unexpected response! serial: 4676, error: 0 [PHONE0]` 같은 modem stale-response 라인의 lowercase `error` 만으로도 매칭됐다. 사용자 표현 그대로 “돌리다 보면 이게 엄청 많이 보여” 였고, 실제 캠페인의 stack_failure 거의 전부가 이 한 패턴이었다.

40분간 단말 logcat 추출해서 RILJ Unexpected 라인 빈도 측정한 결과 **분당 ~0.8 건** 의 background noise. 4번의 burst (각 8건) 가 모뎀의 stale response batch 처리. fuzzer 송신과 무관한 단말 일상 운용 신호.

### 3.1 fix

`oem_ril_error` regex 를 명확한 RIL failure token 만 매칭하도록 좁힘:

```regex
oem[_ -]ril[_ -]error\b
| RILJ:\s*\S*(?:_FAILURE|_FAILED|_ERROR|EXCEPTION|REQUEST_NOT_SUPPORTED|NO_RESOURCES|SECURITY_FAILURE|GENERIC_FAILURE)\b
| SecRIL.*\b(?:Exception|FAILURE|FAILED)\b
| RIL[Dd][_ -]?(?:died|killed|crash|crashed)\b
| rild[_ -]?(?:died|killed|crash|crashed)\b
```

그리고 `rilj_unexpected_response` 라는 별도 warning 패턴 신설 — `RILJ\b.*processResponse:\s*Unexpected response`. verdict 까지 stack_failure 로 안 올라가지만 jsonl 의 `details.adb_warning` 에는 남아 사후 correlation 가능.

`fatal_signal` 카테고리의 SIGSEGV / SIGABRT / tombstone / native_crash / libc_abort / sanitizer / kernel_panic / android_runtime_fatal 등은 그대로 critical 유지. 사용자 요구 (“fatal/crash 는 꼭 잡아줘”) 충족.

## 4. 픽셀 attach — “stuck 처럼 보였는데 idle silent”

오후 늦게 픽셀 10 (Android 16) 으로 fuzz 옮기려고 ADB 연결 + 셀 attach 시도. 한참 진단했는데 결국 **단순 nudge 가 답**이었던 케이스라 따로 적어둘 가치 있다.

### 4.1 증상

- ADB authorize 후 `adb -s ... shell dumpsys telephony.registry` 의 모든 NetworkRegistrationInfo 가 `NOT_REG_OR_SEARCHING`
- `mLteAttachResultType=0`, `getRilVoiceRadioTechnology=0(Unknown)`
- 단말 ip link 에 rmnet 인터페이스 없음 (loopback + dummy0 만)
- `srsenb` 컨테이너 로그의 가장 최근 라인이 **`[RF] Late (detected in asynchronous thread)` 가 22건 burst** 후 더 새 라인 없음 (현재 시각 기준 2분 31초 침묵)

처음에는 srsenb 가 RF Late 누적으로 stuck 됐다고 읽었다. CPU governor 가 powersave 로 다시 돌아간 4-25 시나리오와 닮았어서.

### 4.2 진단

- `cat /sys/devices/system/cpu/cpu0/cpufreq/scaling_governor` → `performance`
- `lsusb -t` 의 USRP B210 → `5000M` (USB 3.0 정상)
- `top` 의 srsenb → CPU 26.7%, RES 700MB, 13시간 가동
- 즉 **CPU/USB/process 모두 정상**, srsenb 는 살아있음

마지막 로그가 침묵인 이유는: 직전에 attached 됐던 A16 이 11:32 detach 한 후로 **활성 user 가 없어서 PUCCH/PUSCH 같은 매 frame 로그가 안 찍힌 것**. RF Late 는 활동이 없을 때의 잔여 burst 로 보였고, cell broadcast (PSS/SSS/MIB/SIB) 자체는 USRP 통해 계속 송출 중.

### 4.3 해결

`adb shell cmd connectivity airplane-mode enable / disable` 로 비행기 모드 toggle → 픽셀 modem 이 cell search 처음부터 다시 → 우리 cell detect → RACH → attach.

attach 후:

| | |
|---|---|
| Cell | LTE Band 3, EARFCN=1600, PCI=1, Open5GS |
| LteAttachResultType | 2 (ATTACHED with IMS APN) |
| UE IP | rmnet1=10.10.10.3 (internet), rmnet2=10.20.20.3 (ims) |
| MME | IMSI `001010000123512` Attach complete (A16 의 ...511 과 다른 SIM) |
| IPsec SA | port_uc=9900, port_us=9901, pcscf port_pc=5109, port_ps=6109 |
| SIP REGISTER | 200 OK, Security-Client esp aes-cbc 협상 완료 |

### 4.4 다음에 같은 증상 보면

- 단말의 `dumpsys telephony.registry` 가 NOT_REG 인데 srsenb 가 silent 면 **무조건 stuck 으로 결론내지 말고** governor / USB / process 살아있음 먼저 확인
- cell broadcast 가 살아있다는 가정 아래 **단말 측 비행기모드 toggle 이 가장 빠른 검증** — 1분 안에 attach 성공하면 srsenb 는 정상이었던 것
- 정말 stuck 이면 RF Late 가 *지속* 발생 (cumulative timestamp 폭주). 단발 burst 후 침묵은 idle.
- 픽셀 10 의 `preferred_network_mode 9,9` (LTE only) 는 setting 적용은 됐어도 carrier config 에 의해 override 될 수 있다는 점은 별개로 확인됨

## 5. 오후 후반 — oracle FP 정리, stale-log 진짜 원인, IPsec SA dispatch fix

§4 의 픽셀 attach 직후 byte_edit_only 캠페인 결과를 분석하면서 추가 청소가 줄줄이 나왔다. 한 묶음으로 정리.

### 5.1 oracle FP 4종 좁힘 (`adb/patterns.py`, `adb/core.py`)

직전 byte_edit_only 캠페인 (`1e89a575`) 의 stack_failure 15건이 모두 같은 종류 FP. regex 가 너무 광범해서 단말 일상 noise 까지 catch:

| 패턴 | 광범 매칭 사례 | 좁힘 후 |
|---|---|---|
| `oem_ril_error` | `RILJ ... error: 0` (case-insensitive 라 lowercase `error` 까지) | 명시적 RIL failure token (`*_FAILURE`/`*_FAILED`/`EXCEPTION`/`GENERIC_FAILURE` 등) 만 critical |
| `native_crash` | CHRE 의 `>>> stationary-unspecified <<<` 같은 sensor 라인 | `>>> .* <<<` 제거. `Fatal signal` + `*** *** ***` boundary marker 만 |
| `nv_corruption` | DSRM-0 의 `network still remains invalid and recovery was not started` (i**nv**alid 의 `nv` 매칭) | `\bNV(?:RAM)?\b` word boundary + `recovery` 제거 |
| `process_died` | Google Maps / Files 같은 일반 앱 종료 | IMS daemon (Samsung/Pixel/AOSP/generic vendor) + RIL + system_server/surfaceflinger 만 |

추가로 **`rilj_unexpected_response`** 라는 새 warning 패턴 분리. `RILJ ... processResponse: Unexpected response` 는 modem 의 routine stale-response batch 처리라 verdict 까지 stack_failure 로 올라가면 안 된다. 다만 jsonl 의 details 에는 남아 사후 correlation 가능.

### 5.2 BLACKLIST_TAGS_EXACT 확장

picsel 캠페인이 catch 한 noise tag 들 (`eikg`, `jqxd`, `dcqy` 같은 obfuscated Google MGS 클래스, `LoadedApk`, `NearbyPresence`, `PickerSyncController`, `TaskFragmentComponent`, `NexusCSStateManager`, ...) + Samsung/Android framework 의 일반 background tag (`DropBoxManagerService`, `WindowManager`, `BatteryStatsService`, `thermal_core`, ...) + CHRE/`WallabyPD` 등을 명시 차단. 약 25개 추가.

**의도적으로 차단 안 한 것**: `NRM-C-0` / `DSRM-0` 같은 modem-side tag. routine warning 과 진짜 baseband fault 가 같은 tag 에서 나오기 때문에 tag 단위 차단 = baseband attack 흔적 losing. critical pattern 좁힘 (5.1) 으로 noise 만 차단되도록 했다.

### 5.3 CVE-class 보강 패턴 4개 추가 (`adb/patterns.py`)

좁힘 작업 중 진짜 IMS / SIP / baseband CVE 흔적이 covered 되는지 audit. 4개 영역이 빠져있었다:

| 패턴 | 매칭 token | severity | category | 의도 |
|---|---|---|---|---|
| `ipsec_sa_failure` | `xfrm.*\b(replay|integrity|seq).*\b(fail|drop|invalid)`, `ESP.*(authentication|integrity|decrypt).*\b(fail|invalid)` | warning | system_anomaly | LTEFuzz / SRsLTE / IMS impersonation PoC 의 ESP 손상 흔적 |
| `ims_forced_deregister` | `IMS_DEREGISTERED_BY_NETWORK`, `deregistered\s+by\s+network`, `forced[_ -]?deregister`, `onDeregistered.*network[_ -]?initiated` | warning | ims_anomaly | IMS state-machine 공격으로 단말이 네트워크 측 강제 deregister 받는 시그널 |
| `aka_auth_failure` | `AKA.*(challenge|sync|sqn|mac|autn).*\b(fail|invalid|mismatch|reject)`, `IK.*(derive|generate).*fail`, `authentication\s+(sync|failure|reject)` | warning | ims_anomaly | 3GPP-AKA SQN desync, malformed AUTN, forged challenge reply |
| `binder_transaction_corrupt` | `Binder\s+transaction\s+(fail|reject)\s+with\s+code`, `TRANSACTION_FAILED.*\bcorrupt`, `unable\s+to\s+unmarshall.*Parcel.*corrupt` | warning | system_anomaly | Android Telephony 의 IPC overflow / heap-spray CVE 흔적 |

`fatal_signal` 카테고리 (SIGSEGV/SIGABRT/tombstone/sanitizer/kernel_panic/android_runtime_fatal/...) 는 그대로 critical 유지. CVE-class coverage 표는 별도로 워크로그 다음 항목 또는 `IMS-IPsec-SA-dispatch.md` 참조.

### 5.4 stale-log scrape 의 진짜 원인 — `logcat -c` 가 main buffer 만 비움

§3 의 RILJ noise 좁힘 후에도 동일 옛날 RILJ 라인이 *매 case 마다 새로* 매칭되는 현상이 있었다 (예: `04-26 10:03:49 serial: 4607` 가 여러 case 에 fresh 처럼 잡힘). oracle scan 의 timestamp floor 부재로 알았으나 더 근본은 `AdbLogCollector.start()` 의 `adb logcat -c` 가 **default (main) buffer 만 비우고** radio / system / crash 는 그대로 두는 것.

새 stream (`adb logcat -b radio`) 시작하면 radio buffer 의 옛 라인을 다시 emit → 첫 _reader_loop 가 fresh 라인으로 인식 → _detector 가 매번 매칭 → matched_line 이 시간상 옛날인데 캠페인 진행 중 case 에 이입. picsel 환경에서는 SIPMSG 가 logcat 에 없어 잘 안 보였지만, A16 환경에서는 매 캠페인마다 동일 옛 라인이 새 stack_failure 만들었다.

Fix:
```python
# 변경 전
subprocess.run(self._connector._adb_cmd("logcat", "-c"), ...)

# 변경 후
for buf in self._config.buffers:
    subprocess.run(self._connector._adb_cmd("logcat", "-b", buf, "-c"), ...)
```

각 buffer 를 명시적으로 비운다. 새 stream 은 진짜 fresh 라인만 받는다. 새 캠페인 (`66aeca76`) 에서 stale FP 0건 확인.

### 5.5 silent strategy registration 청소

`null_byte_only` 추가 시 한 번 fix 했던 silent 0-case 버그를 root 에서 정리. `campaign/core._SUPPORTED_STRATEGIES` 가 `mutator/profile_catalog.SUPPORTED_STRATEGIES_BY_LAYER` 의 하드코드 복사본이었는데, 후자를 import 로 alias:

```python
from volte_mutation_fuzzer.mutator.profile_catalog import (
    SUPPORTED_STRATEGIES_BY_LAYER as _SUPPORTED_STRATEGIES,
    profile_supports_strategy,
)
```

이제 새 strategy 추가는 `profile_catalog.py` + `mutator/core.py` validator 두 곳만 동기화하면 된다. campaign 모드에서 silent drop 재발 안 함.

### 5.6 bypass 모드 응답 수신 fix

§2 에서 잡은 bypass timeout 100% 의 진짜 원인 — campaign 모드는 `sender/cli.py:357-371` 의 single-shot dispatch 를 거치지 않아 `bind_port` 가 누락. campaign 모드도 `null/bypass` 일 때 `bind_port=mt_local_port` (=15100) 를 명시.

```python
# campaign/core.py:_target
self._target = TargetEndpoint(
    ...,
    bind_port=(
        config.mt_local_port
        if config.ipsec_mode in ("null", "bypass")
        else None
    ),
)
```

`container_exec.py` 의 UDP driver 에는 `connect()` + `SO_REUSEPORT` 추가. kamailio 의 wildcard 5060 listener 한테 응답 가로채지지 않도록 4-tuple 매칭 우선.

### 5.7 IPsec SA dispatch fix — server + client SA 양쪽 bind

가장 깊고 중요한 fix. 자세한 내용은 별도 문서 [`IMS-IPsec-SA-dispatch.md`](IMS-IPsec-SA-dispatch.md). 요약:

3GPP IMS IPsec 은 UE 와 P-CSCF 사이 4개 SA (UE/P-CSCF × client/server) 동시 활성. fuzzer 의 `_send_via_native_ipsec` 가 server pair 만 가정하고 한 socket 만 bind 했는데, **rekey 후 일시적으로 client pair 로 routing 되는 시점에 fuzzer 가 응답 못 받음 → timeout**.

picsel 의 `df606d37` 캠페인에서 9/9 timeout (100%) 으로 관찰. wire 분석 결과 단말은 매번 응답 송신, fuzzer 만 못 받음. 5109 (client side P-CSCF port) 로 응답이 오는데 fuzzer 는 6109 (server side) 에만 bind.

Fix: driver 가 두 P-CSCF port (server+client) 둘 다 bind, send 는 primary 로, recv 는 select 로 양쪽 monitor.

```python
# _UDP_DRIVER_SCRIPT
sock_primary = bind(:src_port).connect(:dst_port)         # server pair
sock_alt     = bind(:alt_src_port).connect(:alt_dst_port) # client pair

sock_primary.send(payload)
ready, _, _ = select([sock_primary, sock_alt], [], [], timeout)
if ready: data, peer = ready[0].recvfrom(...)
```

`send_via_native_ipsec` / `_send_via_native_ipsec` / `_send_via_real_ue_direct` 가 alt port pair 를 도출해서 driver 까지 propagate. `port_pc` (UE client port) 와 `port_map[port_pc]` (P-CSCF client port) 를 alt 로 사용.

**기존 동작 보존 보장**: 모든 alt 인자가 default 0. 안 넘기면 driver 의 if 분기로 alt socket 비활성 → single-socket 동작 그대로. TCP path 도 alt 무시. primary bind/connect 의 fatal exit code (2/3) 변경 없음.

검증 (`62bb3c81` 캠페인 직후, 10 case): timeout 100% → **12% (1/8)**. 정상 응답 7건 (모두 500 Server Internal Error — 변이된 OPTIONS 가 단말 SIP stack 자극한 의미 있는 결과).

### IPsec SA rekey 가 뭔가 (간단 요약)

UE 가 정기적으로 (일반적 ~600초) REGISTER 갱신 → 401 challenge → 새 IK/CK → 새 IPsec SA 협상. 이를 rekey. 옛 SA 와 새 SA 가 잠시 공존. 그 공존 구간에 kernel xfrm output policy 가 어느 SA 매칭하느냐는 selector 와 송신 4-tuple 매칭에 따라 다름. fuzzer 의 단일 SA 가정이 깨지는 게 이 시점.

자세한 4-SA 모델 그림과 wire 분석은 `IMS-IPsec-SA-dispatch.md` 참조.

## 6. 남은 청소

- alt-tuple observer event 가 jsonl 의 case record 에 propagate 되는지 확인 필요. 현재 `observer_events.append` 만 하고 있음.
- TCP path 도 dual-bind 적용 여부 검토. 현재는 UDP 만.
- bypass mode 의 `_send_via_container` 도 같은 dual-bind 가 필요한지 (bypass 는 ephemeral port 라 SA selector 매칭 양상 다름).
- oracle stale-log fix 의 또 다른 layer — line 의 logcat timestamp prefix 를 parse 해서 case_start 보다 이전이면 skip 하는 timestamp floor. §5.4 의 buffer-별 `-c` 만으로 cover 안 되는 edge case 보강용.
