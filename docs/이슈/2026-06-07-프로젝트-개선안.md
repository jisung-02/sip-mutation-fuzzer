# 2026-06-07 프로젝트 개선안

대상 프로젝트: `volte-mutation-fuzzer`  
대상 브랜치: `fix/strategy`  
기준 커밋: `2ce53a6 feat: add iphone ims mutation profile`  
추가 작업트리 상태: `iphone_option_tag_negotiation` 관련 uncommitted 변경이 있으며, 현재 focused test는 통과한다. 다만 전체 CI/실기기 검증은 아직 별도 확인이 필요하다.

## 제출 요약

현재 프로젝트는 "단말 지향 VoLTE/IMS SIP mutation fuzzer"라는 방향이 선명하고, 구현도 꽤 멀리 와 있다. `real-ue-direct`가 기본 운용 축이고, `pixel_ims` / `iphone_ims` 프로필까지 들어왔다. 단위/기능 테스트도 강하다.

하지만 지금 바로 개선해야 할 병목은 새 기능 추가가 아니다.

1. CI 품질 게이트가 현재 로컬 기준으로 빨갛다.
2. 운영 문서 일부가 현재 코드와 충돌한다.
3. Pixel/iPhone 전용 프로필은 코드와 테스트는 있지만 실기기 검증 결과물이 아직 부족하다.
4. 장기 퍼징 결과를 "보고서로 제출 가능한 finding bundle"로 만드는 마지막 포장이 약하다.
5. 현재 작업트리에 iPhone option-tag strategy 추가가 진행 중이며, focused test는 통과했지만 전체 품질 게이트와 실기기 검증은 아직 남아 있다.

따라서 다음 개선은 "더 많은 mutation"보다 "신뢰 가능한 실험 산출물" 쪽으로 잡아야 한다. 이 프로젝트의 목적은 데모가 아니라 취약점 징후를 재현 가능한 증거로 남기는 것이다.

## 현재 상태 근거

### 강점

| 항목 | 근거 | 판단 |
| --- | --- | --- |
| 실기기 우선 운영 기준 | `docs/AI_AGENT_GUIDE.md`는 `real-ue-direct`를 현재 기본 기준으로 명시하고, `campaign run` 기본값도 `--target-msisdn 111111`, `--ipsec-mode native`라고 설명한다. | 방향이 맞다. softphone-first로 되돌아가면 안 된다. |
| 프로필 축 분리 | `src/volte_mutation_fuzzer/mutator/profile_catalog.py`에 `legacy`, `delivery_preserving`, `ims_specific`, `parser_breaker`, `pixel_ims`, `iphone_ims`가 중앙 관리된다. | 확장 구조가 좋다. |
| 실기기 기본 관측 | `CampaignConfig`는 `real-ue-direct`에서 process check를 끄고 ADB/pcap을 자동 활성화하며, `pcap_interface`를 `br-volte`로 둔다. | 실제 캠페인 운영에 맞다. |
| 증거 수집 primitive | `EvidenceCollector`가 `crash`, `stack_failure`, `suspicious` 케이스에 대해 `summary.json`, sent payload, response, pcap, ADB/iOS 로그, mutation ops를 묶는다. | finding bundle의 기반은 있다. |
| 기능 테스트 | 최신 dirty worktree 기준 `uv run pytest`는 `952 passed in 49.63s`다. iPhone option-tag focused test도 통과한다. | 기능 회귀 테스트 기반은 좋다. 남은 자동화 병목은 lint/format/type check다. |

### 문제

| 항목 | 근거 | 영향 |
| --- | --- | --- |
| CI가 실제로 통과하지 않는다 | `uv run ruff check .`는 `tests/mutator/test_sdp.py`의 unused import/variable 5건으로 실패한다. `uv run ruff format --check .`는 43개 파일 reformat 필요. `uv run ty check`는 150 diagnostics. | PR 품질 게이트가 신뢰되지 않는다. 새 기능을 올려도 리뷰가 본질보다 포맷/타입 잡음에 묻힌다. |
| CI가 pytest를 실행하지 않는다 | `.github/workflows/ci.yml`은 ruff, format, ty만 실행하고 pytest 단계가 없다. | 가장 강한 검증인 950개 테스트가 GitHub PR에서 빠진다. |
| 로드맵이 현재 코드와 충돌한다 | `docs/발전-로드맵.md`는 ADB/logcat, pcap, stateful dialog fuzzing을 "현재 할 수 없는 것"으로 적지만, 현재 소스에는 `adb/`, `ios/`, `capture/`, `dialog/`, `campaign/evidence.py`, `campaign/dashboard.py`가 있다. | 운영자가 오래된 로드맵을 믿으면 잘못된 작업 순서를 잡는다. |
| 오픈 이슈가 stale이다 | `docs/이슈/오픈-이슈.md`는 ADB logcat을 미구현으로 둔다. 현재는 ADB collector, detector, snapshot path가 구현되어 있다. | 해결된 이슈가 계속 미해결처럼 보인다. |
| iPhone 전용 검증이 미완이다 | `docs/AI_AGENT_GUIDE.md`는 iPhone 전용 MT-INVITE 응답 검증과 iPhone template 부재를 명시한다. | `iphone_ims`가 들어왔어도 "실기기 validated"라고 말할 수 없다. |
| iPhone option-tag 작업이 새로 들어왔다 | 현재 작업트리 diff는 `iphone_option_tag_negotiation`를 profile/default pool/CLI help/test에 추가하고 focused test 2건은 통과한다. | 전체 ruff/format/ty/pytest와 실기기 iPhone baseline으로 검증해야 코드 기준선에 합류할 수 있다. |
| 최신 실기기 결과가 빈약하다 | 최근 `results/20260427_125621_70b892a6/campaign.jsonl`는 header만 있고 case/footer가 없다. | 제출 가능한 실험 결과로 쓰기 어렵다. |

## 개선안 우선순위

### P0. CI를 먼저 녹색으로 만든다

목표: PR을 열었을 때 "현재 프로젝트가 기본 품질선 위에 있다"는 것을 자동으로 증명한다.

작업:

1. 현재 작업트리의 iPhone option-tag strategy 변경을 기준선에 합류시킨다.
   - pytest는 통과했으므로 lint, format, type check와 실기기 검증으로 확장
   - `docs/AI_AGENT_GUIDE.md`와 `docs/USAGE.md`의 iPhone profile 설명이 구현과 같은 strategy 목록을 말하는지 확인
   - 실기기 iPhone identity baseline 후 `iphone_ims` smoke run에 option-tag strategy가 실제로 포함되는지 결과로 확인
2. `ruff check` 실패 5건 정리
   - `tests/mutator/test_sdp.py` unused import 4건 제거
   - unused local variable 1건 제거
3. `ruff format .` 적용 또는 reformat 범위 검토 후 정리
4. `ty check`를 두 단계로 나눈다
   - 1차: `src` production diagnostics 우선 0에 가깝게
   - 2차: tests diagnostics는 narrowing/cast 정책을 정해서 별도 정리
5. GitHub Actions에 pytest 추가
   - 현재 CI: `ruff check`, `ruff format --check`, `ty check`
   - 제안 CI: 위 3개 + `uv run pytest`

완료 기준:

```bash
uv run ruff check .
uv run ruff format --check .
uv run ty check
uv run pytest
```

위 4개가 로컬과 CI에서 모두 통과한다.

기대 효과:

- 이후 개선안의 모든 변경이 같은 기준으로 검증된다.
- "기능은 되는데 CI는 빨간" 상태를 끝낸다.

### P1. Pixel/iPhone 전용 프로필의 실기기 검증 캠페인을 만든다

목표: `pixel_ims`와 `iphone_ims`를 "코드에 있음"에서 "실험으로 설명 가능" 단계로 올린다.

작업:

1. 각 단말 슬롯마다 identity baseline 1 case를 먼저 기록
   - Pixel/Galaxy A17 슬롯: `111111`
   - iPhone 슬롯: `222222`
2. 각 프로필별 최소 검증 캠페인 실행
   - `pixel_ims`: `--pixel --profile pixel_ims --layer wire,byte --strategy default`
   - `iphone_ims`: `--profile iphone_ims --layer wire,byte --strategy default --ios`
3. 결과 디렉터리에 다음을 남긴다
   - `campaign.jsonl`
   - `report.html`
   - `interesting/` evidence
   - pcap
   - ADB 또는 iOS snapshot
   - 실행 로그
   - git sha, command, device mapping, resolver events
4. `docs/이슈/YYYY-MM-DD-real-ue-profile-validation.md`에 결과 요약 기록

완료 기준:

- 각 프로필마다 baseline identity 결과 1개 이상.
- 각 프로필마다 50 case 이상 smoke run.
- `timeout`, `suspicious`, `stack_failure`가 나왔을 때 재현 명령과 pcap/log가 같이 연결된다.

기대 효과:

- `pixel_ims` / `iphone_ims`를 보고서에서 방어 가능하게 설명할 수 있다.
- "새 프로필을 만들었다"가 아니라 "어떤 단말/조건에서 어떤 신호를 봤다"가 된다.

### P2. Finding bundle export를 추가한다

목표: 흥미로운 케이스 하나를 외부 검토자에게 바로 넘길 수 있는 폴더로 만든다.

현재 `EvidenceCollector`는 좋은 시작점이다. 다만 제출물 관점에서는 campaign/run-level context가 더 필요하다.

작업:

1. `fuzzer campaign export-finding <campaign.jsonl> --case-id N` 명령 추가
2. export 결과 구조 제안

```text
finding-case-000123/
  README.md
  summary.json
  sent.sip 또는 sent.bin
  response.sip
  capture.pcap
  adb_log.txt 또는 ios_log.txt
  mutation_ops.txt
  environment.json
  reproduction.sh
```

3. `environment.json`에는 최소한 아래를 포함
   - git sha, branch, dirty status
   - full campaign command
   - target MSISDN, profile, layer, resolved strategy, seed
   - resolver observer events
   - ipsec mode, transport
   - pcap interface
   - device metadata, 가능하면 ADB/iOS device info
4. `README.md`는 사람이 읽는 형태로 자동 생성
   - 무엇을 보냈는가
   - 어떤 응답/이상 징후가 있었는가
   - 어떻게 재현하는가
   - 왜 false positive가 아닐 가능성이 있는가

완료 기준:

- 임의의 `suspicious`/`stack_failure` 케이스에 대해 export 명령이 단일 폴더를 생성한다.
- 폴더만 복사해도 재현 명령과 증거를 이해할 수 있다.

기대 효과:

- 교수/리뷰어/협업자에게 "이 케이스가 왜 중요하지?"를 설명하는 시간이 줄어든다.
- 나중에 CVE급 finding으로 발전할 때 증거 체인이 깨지지 않는다.

### P2. stale endpoint self-heal은 먼저 측정하고 결정한다

목표: real-UE timeout의 원인이 stale endpoint인지 실제 단말 무응답인지 구분한다.

근거:

- 기존 plan 문서에는 stale endpoint self-heal과 Phase 0 측정 계획이 있다.
- 하지만 `docs/이슈/2026-04-28-real-ue-stale-baseline.md` 같은 실제 측정 산출물은 아직 보이지 않는다.
- timeout을 전부 stale로 보면 잘못된 self-heal이 실제 crash/anomaly 신호를 가릴 수 있다.

작업:

1. 50 case baseline run으로 stale marker를 계측
2. timeout case마다 다음을 분류
   - host changed
   - port changed
   - resolver source changed
   - xfrm state absent
   - UE response observed in pcap but app-level timeout
   - genuine no-response
3. stale 비율이 10% 이상이면 one-shot self-heal retry 구현
4. 10% 미만이면 self-heal보다 observer/reporting 개선을 우선

완료 기준:

- `docs/이슈/YYYY-MM-DD-real-ue-stale-baseline.md`에 정량 결과가 남는다.
- 구현 여부가 감이 아니라 측정으로 결정된다.

기대 효과:

- timeout 해석이 훨씬 정직해진다.
- 불필요한 retry가 finding 신호를 덮지 않는다.

### P3. 운영 스크립트는 현재 CLI에 맞춰 다시 만든다

목표: 장기 퍼징을 매번 수작업으로 조립하지 않는다.

작업:

1. `scripts/run_real_ue_validation.sh`
   - identity baseline
   - 50 case profile smoke
   - report generation
   - evidence export
2. `scripts/run_longrun_pixel.sh`
3. `scripts/run_longrun_iphone.sh`
4. 모든 script는 `set -euo pipefail`과 dry-run 출력 지원
5. 구식 `--pcap-dir`처럼 현재 CLI와 맞지 않는 옵션을 문서/스크립트에서 제거

완료 기준:

- 새 서버 세션에서 script 하나로 baseline + smoke + report까지 생성된다.
- script 내부 명령은 `uv run fuzzer campaign run --help`와 불일치하지 않는다.

기대 효과:

- 실험 재현성이 올라간다.
- 문서 복붙보다 운영 실수가 줄어든다.

## 하지 말아야 할 개선

1. softphone-first 로드맵으로 되돌아가지 않는다.
   - 현재 프로젝트 지침은 실기기/`real-ue-direct` 우선이다.
2. MSISDN to IP hardcoded fallback을 되살리지 않는다.
   - 디바이스 회전 환경에서는 silent mis-route가 된다.
3. 모든 `runtime_complete` 메서드를 실기기 validated라고 쓰지 않는다.
   - 현재 real-device baseline은 `INVITE` 중심이다.
4. finding 없이 "대시보드"부터 키우지 않는다.
   - 결과 시각화보다 먼저 증거 품질과 재현성이 필요하다.
