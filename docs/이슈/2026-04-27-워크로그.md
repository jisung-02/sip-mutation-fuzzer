## 1. SDP-aware wire 변이 시리즈 (Stage 1~3)

### 1.1 동기

기존 `--strategy default` (wire) 가 헤더 값에만 변이를 가했다. INVITE 의 SDP body — `m=audio ...`, `c=IN IP4 ...`, `a=rtpmap:96 AMR-WB/16000` 같은 라인 — 은 거의 그대로 통과. SDP/media 스택의 grammar-level edge case (잘못된 codec clock rate, 두 m= 라인, version=99, sendrecv+sendonly conflict, body 바이트 corruption) 가 fuzz target 에서 빠져있었다.

3 단계로 분리:

- **Stage 1 — `sdp_boundary_only`**: SDP 라인 안의 token-level 변이. m= 의 port, transport, c= 의 address, rtpmap 의 codec name / clock rate, b= bandwidth.
- **Stage 2 — `sdp_struct_only`**: line set 변이. `extra_media` (두 번째 m= 추가), `extra_attribute` (a= 추가), `missing_session_name`/`missing_origin` (s=/o= 제거), `version_corrupt` (v=99/A/empty), `direction_conflict` (sendrecv + sendonly).
- **Stage 3 — `sdp_byte_edit`**: body 한정 byte-level surgical edit. 기존 `byte_edit_only` 와 동일한 10 variant (`trim_*`, `flip_*`, `dup_byte`, `swap_adjacent`, `insert_byte`, `insert_null`) 를 SIP framing 은 건드리지 않고 SDP body 만 손대도록 제한.

세 strategy 모두 공통:
- Content-Type 이 `application/sdp` 가 아니거나 body 가 비면 `ValueError` → multi-mutation 루프는 graceful break (try/except 로 흡수).
- mutation 후 Content-Length 자동 재계산.
- 결정성: 같은 seed → 같은 출력. RNG 호출 순서 고정.

### 1.2 Stage 3 우회 — sub-agent 타임아웃과 stash 복원

Stage 3 implementation 만 sub-agent 가 commit 했는데 (`2e755a2`) test 클래스가 빠진 상태로 stream timeout. 직후 `git reset HEAD~1 --soft` 로 commit 롤백했고, working tree 의 변경 분이 stash 에 들어가 있었다.

## 2. `--strategy default` 풀에 SDP 통합

### 2.1 문제

Stage 1~3 추가 후에도 `--strategy default --profile delivery_preserving` 캠페인은 `safe`/`header_whitespace_noise` 두 풀 항목만 픽. 사용자가 “default 일 때 SDP 도 영향 줬으면” 라고 요청.

### 2.2 fix

두 변경:

1. **`PROFILE_DEFAULT_STRATEGY_POOLS` 확장**

```python
"delivery_preserving": {
    "wire": ("safe", "header_whitespace_noise",
             "sdp_boundary_only", "sdp_struct_only", "sdp_byte_edit"),
    ...
},
"ims_specific": {
    "wire": ("safe", "alias_port_desync",
             "sdp_boundary_only", "sdp_struct_only", "sdp_byte_edit"),
    ...
},
```

`parser_breaker` 는 의도적 제외 (Stage 1/2 패턴 동일).

2. **`_is_strategy_applicable` 에 SDP 분기 추가**

```python
if (
    layer == "wire"
    and strategy in {"sdp_boundary_only", "sdp_struct_only", "sdp_byte_edit"}
):
    return (
        editable_message is not None
        and self._has_sdp_body(editable_message)
    )
```

`_has_sdp_body` 는 Content-Type 헤더 값에 `application/sdp` 포함 여부 + body 비어있지 않은지 둘 다 체크.

`_resolve_default_strategy` 는 이미 fallback ring scan 을 가지고 있음 — 처음 픽이 applicable 아니면 pool 을 `(start_index + offset) % len(pool)` 로 돈다. SDP 분기 추가만으로:
- INVITE-with-SDP → 5개 풀 항목 모두 applicable, 자연스럽게 SDP 변이 섞임
- OPTIONS / REGISTER → SDP 항목 자동 skip, `safe` / `header_whitespace_noise` / `alias_port_desync` 로 fallback

### 2.3 깨진 기존 테스트

`tests/mutator/test_core.py` 의 `test_mutate_editable_delivery_preserving_default_wire_keeps_via_intact` 가 seed=1 → strategy=`safe` 라는 specific 한 가정. 풀 길이가 2→5 로 늘어 RNG 결과가 바뀌어 깨짐. 테스트 본질 (Via header 보존) 은 유지하고 strategy assertion 만 `assertIn(case.strategy, {"safe", "header_whitespace_noise"})` 로 완화.

### 2.4 신규 테스트 3건

- `test_delivery_preserving_default_can_pick_sdp_strategies`: 60 seed sweep, 적어도 1건 sdp_* 픽 확인
- `test_ims_specific_default_can_pick_sdp_strategies`: 동일
- `test_default_skips_sdp_when_message_has_no_sdp_body`: OPTIONS + 40 seed sweep, sdp_* 픽 0건 (모두 fallback)

## 3. 부차 작업

### 3.1 multi-round mutation (`15fd268`)

`--mutations-per-case N` (≥2) 지원. 한 case 당 N 회 변이를 stack. mutator/generator/dialog/campaign/CLI 까지 일관 전파. SDP 시리즈가 `try/except ValueError: break` 로 graceful break 를 의존하는 부분의 기반.

### 3.2 mutate CLI 렌더링 강화 (`07ae770`, `5b876b1`)

`uv run fuzzer mutate request INVITE ...` 결과를 보기 좋게: 변이 전후 wire text 양쪽 출력, mutation diff, 영향 받은 컨텍스트와 byte 값 포맷. 디버깅 시 path/before/after JSON 만 보던 시절과 비교해 재현·분석이 빨라졌다.

### 3.3 campaign 에러 분류 (`a5ee17c`)

`CampaignExecutor` 가 예외를 잡을 때 type 과 message 를 모두 jsonl 에 남기도록. 기존엔 message 만 남아 같은 ValueError 안에서 “SDP body required” 와 “alias port not found” 구분이 어려웠다.

### 3.4 real-UE self-heal retry plan (`529f45e`)

xfrm state stale (UE port_pc/port_ps 변경 후 fuzzer cache miss) 시 재등록 후 retry 흐름. 구현은 미진행, plan 문서만 반영. 후속 세션 항목.

### 3.5 real-UE self-heal **phasing** plan + Phase 0 ship (`316592b`, `a3578d9`)

`529f45e` 의 self-heal retry 를 단번에 구현하기 전에, "정말 그게 필요한가" 를 먼저 측정으로 가르기로 함. Phase 0 / Phase 1 로 쪼갠 phasing plan (`docs/superpowers/plans/2026-04-27-real-ue-self-heal-phasing.md`) 작성 후 Phase 0 만 즉시 ship.

#### 발단 — uncoupled lookup

`CampaignExecutor._execute_mt_template_case` 가 `ue_ip` 와 `(port_pc, port_ps)` 를 두 번 별도로 live-resolve 하고 있었다 (`campaign/core.py:1003-1018`):

1. `ue_ip = self._ue_resolver.resolve(...).host`
2. `port_pc, port_ps = self._resolve_ports_live(msisdn)`

두 lookup 사이에 단말 재등록이 끼면 host 는 generation N, port 는 generation N±1 이 되어 — 정확한 머신에 잘못된 protected port pair 로 송신. 단말은 datagram 을 silent drop. fuzzer 입장에선 verdict=`timeout`. "단말 무응답" 과 "uncoupled resolution" 이 구분 안 됨.

`RealUEDirectResolver.resolve_protected_ports(msisdn, *, ue_ip=None)` 는 이미 `ue_ip` kwarg 를 지원하고 있었지만 (`sender/real_ue.py:507-509`), campaign-side wrapper `_resolve_ports_live(msisdn)` 가 그걸 흘려버리고 있었다.

#### Phase 0 — 3 줄짜리 fix

- `316592b` — `tests/campaign/test_core.py:965` 에 forwarding contract 단위 테스트. 사전엔 `TypeError: _resolve_ports_live() got an unexpected keyword argument 'ue_ip'` 로 RED.
- `a3578d9` — `_resolve_ports_live` 시그니처에 `*, ue_ip: str | None = None` 추가, 본문에서 resolver 로 forward, 호출부 (`campaign/core.py:1024-1030`) 에서 이미 line 위에서 resolved 된 `ue_ip` 를 그대로 패스. RED → GREEN.

검증:
- `tests/campaign/test_core.py` 95 passed (신규 1건 포함).
- 광역 회귀 720 passed (719 baseline + 1).
- 기존 `patch.object(executor, "_resolve_ports_live", return_value=(...))` 모킹은 새 keyword 를 통째로 무시하므로 무손상.

#### Phase 1 — 보류 (측정 후 결정)

attempt-trace + payload classifier (endpoint-baked vs endpoint-independent) + same-case retry 인프라는 별도 plan (`docs/superpowers/plans/2026-04-27-real-ue-self-heal-retry.md`, `529f45e` 후 reframe됨) 으로 미루고, **Phase 0 측정 결과** 에 따라 진행 여부 게이팅. 기준:

- 잔여 timeout `< 10%` → Phase 1 보류 (Phase 0 만으로 충분)
- `10~30%` → 축소 진행 (MT 트랙만)
- `> 30%` → 전체 진행

측정 (Phase 0 phasing plan 의 Task 3) 은 testbed 환경 준비 후 다음 세션. `docs/이슈/2026-04-28-real-ue-stale-baseline.md` 에 결과 기록 예정.
