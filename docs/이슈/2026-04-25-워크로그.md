## 1. 잔여 race: kamailio 가 reply 가로채기

어제 native UDP 가 SOCK_DGRAM + xfrm 자동 ESP 로 바뀐 뒤, tcpdump 상에서는 PCSCF→A16 ESP 송신 + A16 응답 ESP 수신이 매 케이스마다 깔끔하게 보였다.  하지만 fuzzer 의 verdict 는 5/5 timeout.

### 1.1 진단

- A16 응답 ESP 가 도착함 (SPI 0x1001, 436~452 byte)
- 커널이 SA 로 복호화 → 평문 UDP datagram (dst=PCSCF:6100)
- PCSCF 컨테이너 안에서 6100 UDP 를 누가 listen 하느냐: **kamailio 와 우리 fuzzer 소켓 둘 다**
- 우리 fuzzer 소켓은 SO_REUSEPORT 로 bind 했고 kamailio 도 같은 옵션으로 listen
- Linux UDP socket lookup 은 SO_REUSEPORT 그룹 내에서 4-tuple hash 기반으로 한 소켓을 결정적으로 픽
- 우리 4-tuple `(10.20.20.2, port_ps) → (172.22.0.21, 6100)` 가 항상 같은 hash 라 항상 같은 소켓에 라우팅 — 그 소켓이 kamailio 였음
- kamailio 가 받은 응답은 자기 transaction 테이블에 매치 안 되니 **stray response 로 조용히 드랍**
- fuzzer 의 `recvfrom()` 는 절대 깨어나지 않음 → `socket.timeout` → 케이스 timeout

`docker logs pcscf` 에 이 stray response 의 흔적이 전혀 안 남는다는 점도 같이 확인했다.

### 1.2 수정 (`f247f36`)

Linux UDP socket lookup 은 **connected socket 을 wildcard-bound listener 보다 우선**한다 (SO_REUSEPORT hash 와 무관).  fuzzer 의 `_UDP_DRIVER_SCRIPT` 가 `bind()` 직후 `connect((dst_ip, dst_port))` 하도록 변경.  그 뒤로는 A16 으로부터 오는 reply 가 우리 connected 소켓에 우선 매칭 → kamailio 는 아예 받지 못함.

`sendto()` → `send()` 로 같이 변경 (connected socket 은 sendto 에 명시적 destination 을 넣을 수 없음).

### 1.3 검증

A16 의 protected port 가 6100/6101 → 8000/8001 로 재등록된 뒤 (자세한 건 §2 참고) 다시 캠페인 돌려서:

```
case=0 verdict=normal code=500 elapsed=728ms
case=1 verdict=normal code=500 elapsed=756ms
case=2 verdict=normal code=500 elapsed=716ms
case=3 verdict=normal code=500 elapsed=732ms
```

normal/500 이 안정적으로 나오는 걸 확인.  500 ("Server Internal Error") 은 A16 IMS 스택이 mutation 으로 깨진 헤더에 일관되게 거부하는 응답으로, 정상적인 fuzzing oracle 출력.

## 2. 두 번째 race: UE 가 재등록하면 캐시된 포트가 stale

§1 과 별개로 발견. 어제 commit `9d77381` 이후 native MESSAGE 경로는 매 케이스마다 live resolve 하도록 정리됐지만, **MT INVITE 템플릿 경로는 여전히 `_resolve_ports_cached`** 로 한 번만 resolve 후 캠페인 끝까지 캐시 사용 중이었다.

### 2.1 증상

UE 가 캠페인 도중 재등록하면:
- xfrm SA 새로 깔리고 protected port 바뀜 (오늘 6100/6101 → 8000/8001 발생)
- fuzzer 는 캐시된 옛날 포트로 계속 송신
- 매 케이스 timeout 으로 직진 (옛 포트엔 listener 가 사라짐)
- 외부에서 보면 "real UE 가 안 응답" 처럼 보임 → 진짜 timeout 케이스와 구분 불가

### 2.2 수정 (`373a1cf`)

- `_cached_ports` 필드 + 캠페인 도중 무효화 로직 제거
- `_resolve_ports_cached` → `_resolve_ports_live` 로 rename (이름이 동작과 맞도록)
- 매 MT 케이스마다 docker logs / xfrm state 신선하게 읽음. 케이스당 ~150ms 추가지만 정확성과 못 바꿈.

캐시는 **MT 경로에만** 있었고 일반 MESSAGE 경로는 어제 이미 해결됐던 것을 본 가지에 맞춰 통일.

## 3. 결과 가시성: 보낸 것도 jsonl 에 박기

native UDP 검증 도중 사용자가 "보낸 / 받은 것 양쪽 평문 다 결과에서 즉시 보이게" 요구.

### 3.1 현 상태 점검

- `case.raw_response`: ✓ A16 응답 평문 SIP 전체 (예: `SIP/2.0 500 Server Internal Error\r\nVia: ...\r\nCall-ID: ...\r\n\r\n`)
- `case.raw_request`: ✗ — 보낸 mutated SIP 평문은 jsonl 어디에도 없음. `reproduction_cmd` 로만 재생성 가능

또한 한 가지 사실 확인: **fuzzer 프로세스 입장에선 양쪽 평문 둘 다 갖고 있다**.  커널 xfrm IN policy 가 ESP 도착 즉시 복호화해서 socket 레이어엔 이미 평문으로 올라오기 때문.  `recvfrom()` 시점이 평문이라 굳이 추가 작업 없어도 보유 상태.  단지 case_result 에 안 박고 있던 것.

### 3.2 수정 (`b388f9f`)

- `CaseResult` 모델에 `raw_request: str | None` 필드 추가 (`raw_response` 옆)
- `_build_case_result` 시그니처에 `raw_request` 추가
- 두 호출부 (일반 send 경로 / MT 템플릿 경로) 가 `_payload_to_text(sent_payload)` 를 넘기도록
- 모듈 상단에 `_payload_to_text(payload: str | bytes | None) -> str | None` 헬퍼 — bytes 는 `errors="replace"` 로 UTF-8 디코드 (binary fuzz payload 도 안전하게 직렬화)

이제 jsonl 한 줄만 보면 양쪽 평문 비교 가능:

```python
print(d['raw_request'])   # mutator 가 만든 wire bytes
print(d['raw_response'])  # A16 의 SIP 응답 전체
```

## 4. pcap 가시성: ESP 도 잡히게

### 4.1 증상

native 캠페인 결과 디렉터리의 `pcap/case_*.pcap` 파일들이 거의 다 24 byte (pcap global header 만, packet record 없음).

### 4.2 원인

`PcapCapture.__init__` 의 기본 BPF 필터:

```python
filter_expr: str = "udp port 5060 or tcp port 5060"
```

native IPsec 트래픽은 **ESP (IP proto 50)** 로 캡슐화돼서 wire 에 나간다.  port 5060 매칭이 안 되니 모든 native 패킷이 필터에서 떨어짐.  bypass/null 모드 (plaintext UDP 5060) 만 의도된 동작이었다.

### 4.3 수정 (`67ac2ed`)

기본 필터를 `"udp port 5060 or tcp port 5060 or esp"` 로 확장.  `esp` 는 BPF/tcpdump 의 키워드로 IP proto 50 을 의미.

- bypass/null/native 모든 모드에서 wire 레벨 캡처 정상화
- MT 경로는 자기 host 기반 필터를 따로 만들어 쓰니 이번 변경에 영향 없음
- ESP payload 자체는 디스크상 여전히 암호문 — Wireshark 에 `ip xfrm state` 출력 SA 키 등록해서 열면 평문 SIP 까지 보임. fuzzer 결과 가시성은 §3 의 jsonl 평문이 1차 채널이고, pcap 은 wire-level 타이밍/순서 검증용 보조 채널.

## 5. 오늘 끝에서 끝까지 검증된 것

A16 (Samsung SM-A165N, MSISDN 111111, IMSI 001010000123511) 대상 native IPsec UDP fuzzing 이 다음 조건을 모두 만족하면서 작동한다는 것이 확인됨:

1. **wire 레벨**: ESP 송신 + A16 의 ESP 응답이 P-CSCF eth0 에 매번 정상 캡처
2. **socket 레벨**: fuzzer 의 connected UDP socket 이 reply 를 직접 수신 (SO_REUSEPORT race 회피)
3. **oracle 레벨**: response code 500 으로 verdict=normal 안정 산출 (mutation 이 깨뜨린 헤더에 대한 A16 의 일관 거부)
4. **결과 인스펙션**: 매 케이스 jsonl 한 줄에 raw_request + raw_response 둘 다 박힘
5. **wire 인스펙션**: per-case pcap 에 ESP 패킷 보존, SA 키로 복호화 가능

지금은 wire pcap 을 평문으로 보려면 매번 손으로 `ip xfrm state` 출력에서 키를 뽑아 Wireshark 에 등록해야 함.  캠페인 시작 시 SA 스냅샷을 결과 디렉터리에 `xfrm_sa.txt` (또는 wireshark esp_sa 포맷) 로 같이 저장하면 더블클릭으로 복호화 가능.  구현 ~30줄.

### 6.2 부하 시 재등록 동작 검증
오늘 검증은 MESSAGE 5케이스 단발성이었음.  100~1000 케이스 장기 캠페인에서 UE 가 자연 재등록하는 경우 §2 의 live resolve 가 매번 신선한 포트를 잡는지, 어느 시점에 재등록 감지가 늦어지는지 모니터링 필요.

### 6.3 TCP 경로
A16 의 single-stream TCP 제약 때문에 native+TCP 는 어제 정리한 그대로 막혀 있음 (commit `b6db2a6` / `720e286` 의 코드는 그대로 남겨둠).  이걸 풀려면 어제 워크로그의 Approach C (kamailio 프록시) 또는 raw TCP injection 이 필요한데, 오늘 native UDP 가 정상화되어서 즉시 우선순위는 낮음.

### 6.4 INVITE / 다른 방법 검증
오늘은 MESSAGE 만 검증.  INVITE / OPTIONS 등 다른 method 도 같은 경로에서 잘 동작하는지 짧게 한 번씩 돌려서 확인 필요.

### 6.5 어제 워크로그의 TODO 와 연결
