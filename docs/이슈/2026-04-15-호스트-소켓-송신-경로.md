# 2026-04-15 · 호스트 소켓에서 A31 까지 — 라우팅과 source IP

## 배경

Slack 에서 나온 질문 정리:

> Q1. 그냥 호스트 소켓으로 쏘면 안 되는 거죠? 리눅스 라우트 설정만 해주면 문제 없을 것 같은데요.
> Q2. 소켓으로 따로 쏘는데 pcap 으로 잡히나요?
> Q3. 라우트 설정을 ogstun2 로 (그 베어러 통해서) 가게끔 하신 거죠?
> Q4. ogstun2 로 왜 가요? 10.20.20.0 이라서?
> Q5. source IP 172.22.0.21 은 어떻게 맞춰요?

이 문서는 위 질문들의 답을 한 곳에 정리한다. 코드는 모두 `src/volte_mutation_fuzzer/` 트리에 있다.

## 한눈에 보는 경로

```
fuzzer (호스트 프로세스)
  │
  │  socket.bind(("172.22.0.21", 15100))   ← source IP spoof
  │  socket.sendto(pkt, ("10.20.20.8", port_pc))
  │
  ▼
호스트 커널 라우팅 테이블
  10.20.20.0/24 via 172.22.0.8             ← setup_ue_route() 가 깔아줌
  │
  ▼
UPF 컨테이너 (172.22.0.8)
  10.20.20.0/24 dev ogstun2                ← UPF 내부 라우트
  │
  ▼
ogstun2 (IMS APN dedicated bearer 의 코어망 종단)
  │
  ▼  GTP-U encapsulation
  │
eNB
  │
  ▼  LTE radio
  │
A31 (UE IP = 10.20.20.8)
```

## Q1. 그냥 호스트 소켓으로 쏘면 되나?

**원리상 됩니다.** 단, 두 가지 전제가 충족돼야 합니다.

1. **라우트** — 호스트 커널이 `10.20.20.8` 같은 UE 대역으로 가는 경로를 알아야 한다. 안 그러면 `No route to host`.
2. **source IP** — P-CSCF 가 받은 SIP 패킷의 source IP 를 보고 "이건 IMS 코어망 안쪽이 보낸 거야" 로 인정해야 한다. 호스트 IP 그대로 나가면 거부됨.

이 두 가지를 자동으로 처리하는 게 fuzzer 의 real-ue-direct 모드.

## Q2. 소켓으로 보냈는데 pcap 에 잡히나?

**잡힙니다.** 이유:

- pcap (libpcap → Linux AF_PACKET / BSD BPF) 은 **인터페이스 레벨**에서 NIC 를 통과하는 프레임을 캡처한다. 송신 프로세스가 누구든, 어떤 socket API 를 썼든 상관없다.
- 우리 fuzzer 도 송신은 평범한 UDP socket, 캡처는 별도 `tshark` 프로세스로 분리해서 동시 수행한다 (`capture/` 모듈).
- 단, 캡처 인터페이스 선택이 중요. `--pcap-interface` 를 안 주면 자동 선택되지만, 호스트→UPF 경로의 `docker0`/bridge 인터페이스를 잡아야 실제 송신 패킷이 보인다.

## Q3 + Q4. 라우트는 ogstun2 로 직접 가나? 왜 10.20.20.0/24 가 ogstun2 로 가나?

**호스트 라우트는 UPF 컨테이너(172.22.0.8) 까지만 보낸다. ogstun2 는 UPF 내부 라우트.**

### 호스트 측 라우트

`infra/core.py:40-77` `setup_ue_route()`:

```bash
# Linux
ip route replace 10.20.20.0/24 via 172.22.0.8

# macOS
route -n add -net 10.20.20.0/24 172.22.0.8
```

기본값 (`infra/core.py:12-14`):

| 상수 | 값 | 환경변수 override |
|---|---|---|
| `_DEFAULT_IMS_SUBNET` | `10.20.20.0/24` | `VMF_REAL_UE_IMS_SUBNET` |
| `_DEFAULT_UPF_IP` | `172.22.0.8` | `VMF_REAL_UE_UPF_IP` |
| `_DEFAULT_PCSCF_IP` | `172.22.0.21` | `VMF_REAL_UE_PCSCF_IP` |

호출 위치: `sender/real_ue.py:644` — 송신 직전에 `RouteCheckResult` 가 fail 이면 자동 setup 후 재확인.

### UPF 내부 라우트

open5gs UPF 는 APN(DNN) 별 TUN 인터페이스를 만든다:

| TUN | APN | 서브넷 |
|---|---|---|
| `ogstun` (또는 `ogstun1`) | `internet` | 보통 `10.45.0.0/16` |
| **`ogstun2`** | **`ims`** | **`10.20.20.0/24`** |

UPF 컨테이너 안에 다음 라우트가 자동으로 생성됨:

```
10.20.20.0/24 dev ogstun2
10.45.0.0/16  dev ogstun
```

### 흐름 요약

1. A31 가 EPS attach + APN=ims 요청 → SMF/UPF 가 `10.20.20.0/24` 풀에서 IP 할당 (예 `10.20.20.8`)
2. 동시에 IMS APN 의 dedicated bearer (시그널링용 QCI=5) 수립, GTP-U 종단점이 UPF 의 `ogstun2`
3. fuzzer 가 보낸 패킷이 호스트→UPF 컨테이너 도착
4. UPF 커널이 dst=`10.20.20.8` 보고 `ogstun2` 로 라우팅
5. open5gs UPF 프로세스가 `ogstun2` TUN 에서 패킷을 read → GTP-U 헤더 씌워서 eNB → UE

→ 결과적으로 베어러 타고 가는 게 맞고, 우리는 그 시작점인 **"호스트가 UE 대역을 모르는 문제"** 만 정적 라우트로 풀어준 것.

UPF 가 죽으면 이 경로가 즉시 단절됨 (`docs/이슈/A31-real-ue-direct-시스템-요구사항.md:37` 참조).

## Q5. source IP 172.22.0.21 은 어떻게 맞춰?

P-CSCF (172.22.0.21) 는 외부 호스트 IP 에서 들어온 SIP 를 거부한다. fuzzer 가 P-CSCF 인 척 위장해서 보내야 함. 두 가지 방법:

### 방법 1 — 호스트에서 IP spoof bind (현재 기본 경로)

`sender/core.py:399-472` `_send_with_spoofed_source()`:

```python
sock = socket.socket(AF_INET, SOCK_DGRAM)
sock.bind(("172.22.0.21", 15100))         # 호스트 NIC 에 없는 IP
sock.sendto(payload, (ue_ip, port_pc))
```

호스트에 없는 IP 에 bind 하려면 커널 옵션 필요:

```bash
sudo sysctl -w net.ipv4.ip_nonlocal_bind=1
```

이게 켜지면 커널이 "내 IP 아닌 172.22.0.21 로 bind 해도 통과" 시켜주고, IP 헤더 source 가 그대로 172.22.0.21 로 나간다. 응답은 라우팅상 호스트로 돌아오므로 같은 소켓에서 받는다.

`target.source_ip` 가 설정되면 이 경로 (`sender/core.py:327-340`). `--ipsec-mode null` 이 이걸 자동 설정.

### 방법 2 — 컨테이너 netns 안에서 송신 (호환 fallback)

`sender/container_exec.py:106-156` `send_via_container()`:

```bash
docker exec -i pcscf python3 -c <driver_script> \
    172.22.0.21 15100 <ue_ip> <port_pc> UDP <timeout> 1
```

P-CSCF 컨테이너의 netns 안에서는 172.22.0.21 이 진짜 로컬 IP 라 평범한 `bind()` 가 통한다. `ip_nonlocal_bind` 를 못 켜는 환경에서의 fallback.

`bind_container=pcscf` + `source_ip=None` 이면 이쪽 (`sender/core.py:289`). `--ipsec-mode bypass` 가 이걸 설정.

### 어느 모드가 어느 경로를 쓰나

| `--ipsec-mode` | source IP 설정 방법 | 경로 코드 |
|---|---|---|
| `null` | 호스트 spoof bind | `_send_with_spoofed_source` |
| `bypass` | docker exec | `_send_via_container` |

## 응답 수신

응답이 P-CSCF→fuzzer 로 돌아오는 경로:

- **방법 1**: docker bridge `172.22.0.0/16` 가 호스트 인터페이스에 직접 붙어 있어, P-CSCF 가 172.22.0.21 로 보낸 응답이 자연스럽게 호스트로 라우팅. 같은 spoof-bind 소켓에서 `recvfrom()`.
- **방법 2**: docker bridge 위에서 응답이 같은 컨테이너로 돌아오므로 driver script 가 그 자리에서 read.

둘 다 **Via sent-by 의 host:port** 를 송신 socket 의 bind 와 일치시켜야 응답을 받는다 (`prepare_real_ue_direct_payload`).

## 명령줄 요약

라우트 수동 설정:

```bash
uv run fuzzer infra setup-route                 # 자동 (기본값 사용)
uv run fuzzer infra setup-route --ims-subnet 10.20.20.0/24 --upf-ip 172.22.0.8
uv run fuzzer infra check-route                 # 현 상태 확인
```

A31 ring 1 회 (라우트는 송신 직전 자동 확인/설정):

```bash
sudo sysctl -w net.ipv4.ip_nonlocal_bind=1      # 방법 1 쓸 때 1 회
uv run fuzzer campaign run \
  --mode real-ue-direct --target-msisdn 111111 \
  --mt --ipsec-mode null \
  --methods INVITE --layer wire --strategy identity \
  --timeout 6 --max-cases 1
```

`--ipsec-mode bypass` 를 쓰면 `ip_nonlocal_bind` 불필요.
