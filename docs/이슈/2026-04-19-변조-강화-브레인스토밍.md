# 2026-04-19 변조 강화 브레인스토밍

이 문서는 현재 대화에서 정리된 **mutation hardening 아이디어**만 따로 모아 둔 메모다.
아직 구현 확정본이나 상세 설계 문서는 아니며, 현재 코드 상태를 기준으로
"어디를 더 공격적으로 바꿀 수 있는가"를 정리한 논의 기록이다.

우선순위 기준은 프로젝트 전체 지침과 동일하게 **`real-ue-direct` + 실기기 경로 우선**이다.

---

## 1. 현재 변조 방식 요약

현재 mutator는 다음 흐름으로 동작한다.

1. `Generator`가 baseline `SIPRequest` / `SIPResponse`를 만든다.
2. `MutationConfig`로 `seed`, `strategy`, `layer`, `max_operations`를 정한다.
3. `model`, `wire`, `byte` 중 한 계층에서 변조한다.
4. 결과는 원본 패킷, 변조 결과, mutation record, before/after를 포함해 반환한다.

핵심 특징:

- 같은 baseline과 같은 `seed`를 쓰면 동일 결과를 재현할 수 있다.
- `max_operations` 기본값이 `1`이라 현재 기본 흐름은 **보통 한 위치만 바뀌는 단일 변조**에 가깝다.
- 일반 packet 경로와 MT-template 경로가 서로 다른 방식으로 mutator에 들어간다.

---

## 2. MT 경로와 일반 경로 차이

### 2-1. 일반 경로

일반 경로는 `Generator`가 SIP catalog + Pydantic 모델 기반으로 만든 baseline packet을 입력으로 사용한다.

특징:

- `PacketModel`이 존재한다.
- `model / wire / byte` 3계층 변조를 모두 사용할 수 있다.
- 구조적으로 "정상 SIP"를 먼저 만들고, 그 위를 변형하는 흐름이다.

### 2-2. MT 경로

MT는 `Mobile Terminated` 의미로, 실제 UE가 수신하는 방향의 패킷을 뜻한다.

현재 MT 경로는:

1. 3GPP/A31 스타일 wire template를 렌더링한다.
2. 그 결과를 `EditableSIPMessage`로 파싱한다.
3. `mutate_editable(...)`로 `wire` 또는 `byte` 변조를 적용한다.

특징:

- full `PacketModel`이 아니라 **실기기 수신용 wire template**가 출발점이다.
- `model` 계층을 넣어도 실제로는 `wire` 쪽으로 내려간다.
- `P-*`, `Record-Route`, `alias`, SDP precondition 같은 **IMS/3GPP surface**를 다루기 쉽다.

실기기 A31 기준으로는 일반 경로보다 MT 경로가 훨씬 중요하다.
일반 SIP 문법만 맞는 패킷은 IMS 앱 레이어에서 조용히 버려질 수 있기 때문이다.

---

## 3. 현재 구현의 한계

현재 구현은 재현성과 분리 구조는 좋지만, 공격성 측면에서는 아직 보수적인 부분이 있다.

### 3-1. 단일 연산 중심

- 기본 `max_operations=1`
- 캠페인 기본 흐름도 대부분 단일 mutation으로 수렴

문제:

- 실제 parser/state bug는 **상관된 두세 개의 모순**에서 터질 가능성이 크다.
- 예: `Content-Length mismatch` 하나보다
  `Content-Length mismatch + body truncation + duplicate header`가 더 위험하다.

### 3-2. 캠페인에서 쓰는 전략 surface가 좁음

mutator 서비스 레벨은 `safe`, `header_targeted` 같은 전략도 지원하지만,
campaign 조합 생성은 현재 사실상 아래만 기본 사용한다.

- model: `default`, `state_breaker`
- wire: `default`, `identity`
- byte: `default`, `identity`

즉, mutator core가 가진 공격성을 campaign이 아직 충분히 사용하지 못하고 있다.

### 3-3. 실기기용 IMS surface 집중도가 낮음

현재도 wire/byte mutation은 가능하지만,
실기기 `real-ue-direct` 기준으로 특히 위험한 아래 영역에 대한 **의도적 전략 이름과 연산 묶음**은 아직 부족하다.

- `P-Asserted-Identity`
- `P-Access-Network-Info`
- `Record-Route`
- `Contact alias`
- precondition SDP
- `Security-*`

## 4. 변조 강화 목표

논의 기준에서 mutation 강화의 목표는 단순히 "더 랜덤하게"가 아니다.

목표는 다음 세 가지를 분리해서 다루는 것이다.

### 4-1. Deliverable but toxic

의미:

- UE까지 최대한 도달하도록 바깥 껍질은 유지
- 내부 상태/의미만 독하게 꼬기

목적:

- 실기기에서 timeout만 양산하지 않고
- 실제 app-layer 처리까지 밀어 넣으면서 crash/logic bug를 노리기

### 4-2. Parser-breaker

의미:

- framing, duplication, truncation, CRLF damage, invalid bytes를 강하게 넣기

목적:

- low-level parser robustness와 boundary handling을 직접 자극하기

주의:

- 실기기 경로에서는 timeout 비율이 크게 늘 수 있다.

### 4-3. IMS-private abuse

의미:

- 3GPP/IMS 특화 필드를 집중 공략

목적:

- 범용 SIP parser보다 실제 carrier/IMS stack의 특수 경로를 노리기

---

## 5. 추천 방향

현재 프로젝트 기준 추천 순서는 다음과 같다.

1. `Delivery-preserving critical`
2. `IMS-specific critical`
3. `Parser-breaker`

이 순서를 추천한 이유:

- 현재 프로젝트의 우선 경로가 `real-ue-direct`
- 목표 타깃이 실기기 UE
- 따라서 "단말에 실제로 도달하는 독성 패킷"을 먼저 늘리는 편이 효율적

즉, 먼저 `A + C`를 만들고, 이후 `B`를 escalation 단계로 붙이는 전략이 적합하다.

## 6. 제안한 전략 패밀리

아래는 현재 대화에서 제안된 mutation 강화용 전략 이름들이다.
아직 코드에 구현된 것은 아니며, **추가 전략 후보**로 정리한 것이다.

### 6-1. `dialog_desync`

핵심 아이디어:

- dialog / transaction 상관관계 필드 간 모순 만들기

예시:

- `To tag` drift
- `From tag` drift
- `CSeq` drift
- `Via branch` drift
- `Contact alias` drift

목표:

- state machine desync
- transaction/dialog matching 오류

### 6-2. `ims_identity_poison`

핵심 아이디어:

- IMS identity 계층을 서로 안 맞게 만들기

예시:

- `P-Asserted-Identity`와 `From` 불일치
- `TEL URI`, `IMPI`, `Contact user` 모순
- originating/terminating identity 충돌

목표:

- identity normalization
- trusted-network identity 처리 경로 자극

### 6-3. `route_shadow`

핵심 아이디어:

- routing chain을 논리적으로 이상하게 만들기

예시:

- `Record-Route` 순서 뒤집기
- 중복 route 삽입
- route/alias 파라미터 충돌
- loose routing 형태 왜곡

목표:

- route set 계산 오류
- remote target/next hop 계산 혼선

### 6-4. `sdp_precondition_conflict`

핵심 아이디어:

- SIP 본문 쪽 SDP 의미를 충돌시키기

예시:

- `m=` 포트와 `a=rtcp` 포트 불일치
- codec/PT 선언 모순
- `curr` / `des` precondition 상충
- media 방향성 충돌

목표:

- SIP parser 이후 단계
- IMS call setup / media negotiation 처리 로직 자극

### 6-5. `framing_desync`

핵심 아이디어:

- parser가 메시지 경계를 헷갈리게 만들기

예시:

- duplicate `Content-Length` with conflicting values
- body truncation
- extra blank line
- folded header abuse
- header/body 경계 어긋남

목표:

- framing parser
- header parser
- payload slicing 경계 조건 자극

### 6-6. `byte_poison`

핵심 아이디어:

- low-level bytes를 더 악랄하게 손상시키기

예시:

- NUL 삽입
- high-bit byte 삽입
- delimiter 반파괴
- 특수문자 인접 truncation
- UTF-8 decode failure 유도

목표:

- byte parser
- decoding / sanitation
- boundary handling 자극

## 7. 핵심 설계 제안: 단일 랜덤 op에서 bundle 기반으로

이번 대화에서 가장 중요하게 나온 제안은
**"랜덤 연산 N개"보다 "의미 있는 공격 bundle 1개"가 낫다**는 점이다.

현재:

- 보통 한 target를 골라
- 한 operator를 적용하는 구조

개선 방향:

- 연관 있는 2~4개 연산을 묶어 하나의 공격 bundle로 정의
- 각 bundle 안에서는 상관된 위치만 함께 바꾼다

예시:

- `Content-Length mismatch + body truncation`
- `To-tag drift + Contact alias drift`
- `Record-Route reorder + Via branch drift`
- `P-Asserted-Identity mismatch + From mismatch`

장점:

- 단순 random보다 훨씬 크리티컬하다
- 재현성 유지가 쉽다
- crash가 났을 때 분석도 쉽다

## 8. real-ue-direct 기준 우선 확장 포인트

실기기 기준으로는 다음이 먼저 필요하다고 보았다.

### 8-1. MT-template 전용 타깃 추가

다음 헤더/필드를 의도적으로 직접 겨냥할 수 있어야 한다.

- `P-Asserted-Identity`
- `P-Access-Network-Info`
- `Record-Route`
- `Contact`의 `alias=...`
- `Supported`, `Require`, `Security-*`
- precondition SDP 필드들

### 8-2. 단계형 공격 프로필

campaign 쪽에서 아래 같은 profile을 따로 둘 수 있다.

- `arrival-safe`
  설명: 도달성 유지 우선, outer shell 보존

- `state-poison`
  설명: dialog / IMS state를 집중적으로 교란

- `parser-kill`
  설명: framing/byte damage를 더 과격하게 수행

### 8-3. oracle 피드백 기반 escalation

예상 아이디어:

- `100/180`까지 도달한 seed 주변은 더 공격적으로 확장
- 전부 timeout이면 outer shell 변조를 줄이고 inner mutation 위주로 조정

즉, 도달성 신호가 보이는 케이스를 더 깊게 파는 방식이다.

---

## 9. 당시 제안한 실제 작업 순서

현재 대화 기준으로는 아래 순서를 먼저 권장했다.

1. campaign이 새 전략 이름을 실제로 스케줄할 수 있게 strategy surface 확장
2. MT wire 전용 IMS/3GPP operator 추가
3. 단일 random op 중심에서 bundle mutation 구조 추가
4. `arrival-safe`, `state-poison`, `parser-kill` 같은 실험 프로필 도입

즉, 단순히 mutator core 연산만 늘리는 것이 아니라
**campaign에서 그 전략을 제대로 돌릴 수 있게 만드는 것**까지 포함해야 한다는 판단이다.

---

## 10. 현재 문서의 범위

이 문서는 다음까지만 반영한다.

- 현재 mutator 구조 해설
- MT와 일반 경로 차이
- mutation 강화를 위한 방향성
- 새 전략 패밀리 후보
- bundle mutation 아이디어
- real-ue-direct 우선 기준

아직 포함하지 않는 것:

- 구현 상세 설계
- 파일별 수정 계획
- 테스트 전략 상세화
- 실제 CLI 옵션 확정
- 정식 spec / implementation plan

필요하면 다음 단계에서 이 문서를 바탕으로
정식 설계 문서나 구현 계획 문서로 승격할 수 있다.
