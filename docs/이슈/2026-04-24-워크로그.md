## 1. 발단: Pixel / A16 이 테스트 셀에 안 붙음

처음 상황은 "새 SIM 을 Pixel 에 꽂았는데 attach 가 안 된다". 서버 상태를 훑으면서 다음을 확인했다.

- Docker container 전부 정상. 모두 4분 전에 올라와 있었음 (서버 재부팅 직후)
- `docker exec pcscf ip xfrm state` — UE IPsec SA **0개**. 어떤 UE 도 지금 IMS 에 등록되어 있지 않음
- MME 로그상 IMSI 001010000123512 (과거 Pixel) 이 04/15 에 attach 성공했던 이력 있음 (LTE 는 잘 붙음)
- Pixel 의 `dumpsys telephony.registry`: `mIsEmergencyOnly=true`, `MCC=450 MNC=06 (LG U+)` — **실제 LG U+ 매크로 셀에 camping 중**, 테스트 셀은 아예 감지 못함
- Pixel carrier config: test001 PLMN 에 대해 `carrier_volte_available_bool = false` — Pixel 은 unknown PLMN 에 대해 VoLTE 를 자동 enable 안 함

그 후 사용자가 A16 (SM-A165N) 으로 교체. A16 도 동일하게 LG U+ 에 camping 되고 있었음.

### 1.1 일차적 진단: RF 도달 부족

srsenb 설정은 `mcc=001 mnc=01, dl_earfcn=1600 (Band 3), n_prb=50, tx_gain=80`.  A16 dumpsys 에서는 scan 목록에 Band 1/5/7 만 뜨고 **Band 3 EARFCN 1600 은 아예 안 잡힘**.  LG U+ 매크로 (Band 1 2100MHz, EARFCN=100) 가 훨씬 강해서 테스트 셀은 깊이 묻힘.

→ **A16 을 USRP 안테나 옆에 물리적으로 근접시킴**.  dumpsys 에 `mOperatorAlphaLong=Open5GS, mMcc=001 mMnc=01` 가 뜨기 시작 — 감지는 됨.  근데 여전히 attach 안 함.

## 2. 진짜 원인: srsenb Late/Overflow + CPU governor = powersave

### 2.1 srsenb 상태가 엉망

srsenb 로그에 `[RF] Late (detected in asynchronous thread)` 수천 줄 연속, 이어서 `[RF] Overflow`.  USRP→host 간 데이터 전송이 밀리는 증상.

첫 의심은 USB 대역폭.  `lsusb -t` 결과 B210 이 Bus 003 Port 1 (480M = USB 2.0) 에 붙어있음. 서버에는 Bus 02/04/06 에 USB 3.0 (10000M) 포트가 비어있었음. 사용자가 케이블을 계속 옮기는 와중에 srsenb 가 **SIGABRT with stack trace in `libsrsran_rf_uhd.so`** 로 크래시.  USRP 뽑으면서 UHD handle 이 꼬인 것. 이건 우리 쪽 원인.

USRP 를 안정적인 포트에 두고 `docker start srsenb` 로 재기동 후, 실행 로그에 다음 경고가 찍혀 있었다:

```
WARNING: cpu0 scaling governor is not set to performance mode.
         Realtime processing could be compromised.
```

**서버 재부팅 후 CPU scaling governor 가 기본값(`powersave`) 로 돌아가 있었다**.  DSP 스레드가 실시간 스케줄링을 못 받아 Late/Overflow 가 터지고, srsenb 가 UE RACH preamble 디코딩을 못 해서 attach 가 성립하지 않은 것.  USB 2.0 은 A31 시절에도 같은 설정으로 잘 돌던 사실이 방증이었음 ("원래는 동작했어").

### 2.2 조치

```
echo performance | sudo tee /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor
```

즉시 적용하고 A16 의 비행기모드 토글. 그 다음 srsenb 로그:

```
[PHY] PRACH: cc=0, 0/1, preamble=46, offset=1.0 us, peak2avg=456.0
[MAC] SCHED: Added user rnti=0x47
[RRC] Rx SRB0 PDU, rnti=0x47 - rrcConnectionRequest
[RRC] Tx SRB0 PDU, rnti=0x47 - rrcConnectionSetup
...
[S1AP] Procedure InitialContextSetupRequest - Received RRC reconf complete
[GTPU] Tx S1-U SDU, UL > eps-BearerID=5, IPv4 10.10.10.2 > 8.8.8.8
```

**A16 이 첫 성공 RACH → RRC → attach → IP 10.10.10.2 할당 → 실제 인터넷 트래픽** 까지 한 번에 됐음.  SNR 30 dB, CQI 15 로 신호도 튼튼.

### 2.3 `setup_host.sh` 에 반영 (commit `dc0d443`)

수동으로 매번 echo 하기엔 재부팅마다 재적용이 필요하니 `setup_host.sh` 에 영구화 로직을 추가했다:

- 새 `setup_cpu_governor()` 함수: 즉시 `/sys/devices/system/cpu/*/cpufreq/scaling_governor` 에 `performance` 를 쓰고, `volte-cpu-performance.service` systemd unit 으로 재부팅 후에도 자동 적용되게 설치
- `setup_all` 에서 단계 5/6 으로 호출
- 독립 옵션 `--cpu` 도 추가
- `--check` 출력에 현재 governor 상태 표시
- 기존 단계 번호 `[N/5]` → `[N/6]` 재넘버

## 3. 첫 campaign 시도: protected port resolver 실패

A16 이 attach 되고 나서 첫 퍼징 캠페인:

```
uv run fuzzer campaign run --mode real-ue-direct --target-msisdn 111111 \
  --methods MESSAGE --ipsec-mode null --profile delivery_preserving ...
```

전부 "send error: could not resolve protected ports for UE via container 'pcscf'" 로 실패.

### 3.1 원인

`src/volte_mutation_fuzzer/sender/real_ue.py` 의 `resolve_ue_protected_ports` 는 P-CSCF 로그/xfrm state 에서 port_pc/port_ps 후보를 추출한 뒤 **UE IP 로 필터** 한다.  그 UE IP 를 어디서 구하냐 보니:

```python
requested_ue_ip = resolve_ue_ip_from_msisdn(msisdn, env=env)
```

`resolve_ue_ip_from_msisdn` 은 **하드코딩된 `_DEFAULT_MSISDN_TO_IP` 테이블** + `VMF_MSISDN_TO_IP_<msisdn>` env override 만 참조.  테이블 값은 `111111 → 10.20.20.8`.

그런데 오늘 서버 재부팅 후 open5gs UE IP pool 맨 앞 자리로 A16 이 **10.20.20.2** 를 받았음.  resolver 가 10.20.20.8 로 필터를 걸면 모든 후보가 날아가고 빈 결과 → `RealUEDirectResolutionError`.

docs 는 "MSISDN → UE IP 자동 resolve" 라고 광고하고 있었는데, 실제로는 host resolver (`RealUEDirectResolver.resolve()`) 만 live 조회였고 port resolver 는 하드코딩 테이블에 묶여 있는 **반쪽짜리 자동화**였다.

### 3.2 수정 (commit `9d77381`)

host 쪽 resolver 는 이미 kamctl → S-CSCF MySQL → P-CSCF logs → OPTIONS ping → xfrm state 순서의 live lookup 체인을 갖고 있다.  그 체인만 재사용하면 됨.

- `RealUEDirectResolver.resolve()` 에 인라인으로 있던 체인을 `_lookup_ue_contact(msisdn, impi)` 로 추출 (non-raising)
- `resolve_protected_ports(msisdn, ue_ip=None)` 메서드에 `ue_ip` 파라미터 추가. 없으면 `_lookup_ue_contact` 로 live 조회, 있으면 그대로 사용
- 모듈 함수 `resolve_ue_protected_ports(..., ue_ip=None)` 에도 동일 파라미터.  주어지면 하드코딩 fallback 을 타지 않음
- `sender/core.py`, `sender/cli.py` 에서는 이미 바로 위 라인에서 `resolver.resolve(target)` 해서 `resolved.host` 가 있으니 그걸 `ue_ip=...` 로 넘겨서 **중복 조회 회피**
- `campaign/core.py:_resolve_ports_cached` 는 msisdn 만 있어서 자동으로 live 조회 경로 타게 됨

테스트 영향: `test_real_ue.py` 의 두 테스트가 `mock_run.assert_called_once_with` 로 subprocess 호출을 1개만 기대하고 있었는데, 이제는 live lookup 체인 때문에 여러 번 불린다. 테스트 의도를 살리기 위해 `resolve_protected_ports("111111", ue_ip="10.20.20.8")` 로 **명시 전달** 하도록 바꿔서 기존 필터 로직 단독으로 검증되게 유지.

743 → 743 pass (`test_core.py` 의 assert_called_once_with 도 새 키워드 반영).

## 4. 두번째 campaign: 10 연속 timeout → circuit breaker

resolver 는 풀렸고, 다시 돌렸는데 이번엔 20 케이스 중 10 연속 timeout → circuit breaker trip (`--circuit-breaker-threshold 10` 기본값) → `status=aborted`.

P-CSCF 쪽 상태를 보니:

```
OPTIONS to sip:001010000123511@10.20.20.2:6100 via sip:10.20.20.2:6100;transport=tcp ... completed with code: 200, Type 1
```

**A16 은 IMS 를 TCP 로 등록했고 kamailio 의 NATPING (TCP OPTIONS) 에는 멀쩡히 200 응답**.  반면 fuzzer 는 `transport=UDP` 로 쏨.  A16 은 UDP 쪽을 listener 자체 없거나 정책으로 드랍. → 전부 타임아웃.

### 4.1 Validator 가 TCP 를 막음

`--transport TCP` 를 붙이니:

```
Configuration error: native ipsec_mode requires transport='UDP'
```

`campaign/contracts.py:226` 에 `native + UDP only` 하드 enforcement 가 있었음. 구조적으로 막힌 건 아니고 단지 validator 보수 설정.

### 4.2 TCP 지원 추가 (commit `b6db2a6`)

- **`campaign/contracts.py`**: `native` 가 UDP/TCP 둘 다 수락하도록 조건 완화
- **`sender/ipsec_native.py`**:
  - 기존 UDP raw driver 옆에 `_TCP_DRIVER_SCRIPT` 신설 — SOCK_STREAM + SO_REUSEPORT + bind + connect + sendall.  커널 xfrm SA 가 있으면 자동으로 ESP 적용되니 복잡한 수동 헤더 작업 불필요
  - `preflight_native_ipsec_target`, `send_via_native_ipsec` 에 `transport: Literal["UDP", "TCP"] = "UDP"` 파라미터 추가.  기본값이 UDP 라 기존 caller 는 무변경
  - observer event: `native-ipsec:send:ok` 는 유지 + `native-ipsec:send:transport:<udp|tcp>` 를 병렬로 추가.  기존 test assertion 호환
- **`sender/core.py`**: `target.transport` 를 preflight/send 로 전달
- 테스트: 기존 "native+TCP 거부" 테스트를 "native+TCP 수락" 테스트로 반전, SCTP 등 다른 transport 는 여전히 거부하는 negative case 추가

## 5. TCP campaign → `EADDRNOTAVAIL` → `port_ps` 스위치 (commit `720e286`)

```
native IPsec injector failed: ... line 32, in <module>
OSError: [Errno 99] Cannot assign requested address
```

line 32 는 `sock.connect((dst_ip, dst_port))`.  bind 는 성공, connect 에서 터짐.

P-CSCF 내부 상태:

```
ESTAB 172.22.0.21:5100  10.20.20.2:6100  users:(("kamailio",pid=141,fd=159))
```

**kamailio 가 이미 `(pcscf:5100 ↔ ue:6100)` 4-tuple 의 TCP 를 ESTAB 로 보유 중**.  같은 4-tuple 로 새 SOCK_STREAM connect() 를 시도하면 커널이 `EADDRNOTAVAIL` 로 거부 (SO_REUSEPORT 와 무관한, 4-tuple 유일성 제약).

### 5.1 진짜 답: destination port 를 port_ps (6101) 로

kamailio 의 ESTAB 은 `port_pc (6100)` 에 물려있음.  하지만 3GPP IPsec 용어로 **P-CSCF 가 UE 로 요청을 보낼 때는 UE 의 서버 listener 인 port_ps 로 보내는 게 맞음**.  port_pc 는 UE 가 자기 요청을 내보내는 쪽이고 응답이 돌아오는 곳.

xfrm OUT policy 리스트에 이미 `src 172.22.0.21 sport 5100 dst 10.20.20.2 dport 6101 reqid 16065` 가 설치되어 있음 → 커널이 그 4-tuple 도 ESP 로 감싸줄 준비가 됨.

`sender/core.py` 에서 한 줄:

```python
resolved_port = port_ps if target.transport == "TCP" else port_pc
```

UDP 는 A31 호환성 때문에 `port_pc` 유지. TCP 만 `port_ps` 로 스위치.

## 6. `ConnectionRefused` → A16 은 single-stream TCP

`port_ps` 로 바꾸니 `EADDRNOTAVAIL` 은 풀렸는데 이번엔:

```
ConnectionRefused: [Errno 111] Connection refused
```

즉 A16 이 **6101 포트에 TCP 리스너 자체가 없어서 RST 로 응답**.  확인:

```
docker exec pcscf ss -tnp | grep 10.20.20.2
ESTAB 172.22.0.21:5100  10.20.20.2:6100  (only 1 connection)
```

A16 의 IMS 스택이 3GPP IMS over TCP 의 **single-stream** 패턴을 쓴다는 뜻. UE 가 P-CSCF 로 **클라이언트 연결 하나만** 열고, P-CSCF→UE 요청도 그 stream 위에 다 실어 내려감. port_ps 에는 아무 TCP server 도 없음.  어떤 새 TCP 연결 시도도 A16 에선 RST.

### 6.1 실제로 쓸 만한 경로 플랜

이 시점에 사용자가 "1번 (raw TCP injection) 플랜 써봐" 라고 해서 3가지를 비교했다:

- **A. 새 raw TCP 3WHS**: SYN → RST (UE 리스너 없음) — **A16 기준 불가**
- **B. kamailio ESTAB 에 in-stream 주입 (`TCP_REPAIR`)**: 기술적으로 가능하지만 kamailio 의 TCP state 와 race/오염 위험. 매 iteration 마다 UE 재등록 필요해질 수 있음
- **C. kamailio 자체를 프록시로 (xhttp/event_route)**: 깨끗하지만 P-CSCF 이미지/설정 patch 필요 (~3일)

→ SSH 로 서버에서 전제 검증:
- `ss -tnp`: kamailio ESTAB 단 하나 (single-stream 확정)
- `docker inspect pcscf`: **privileged + CAP_NET_ADMIN** → A/B/C 모두 기술적으로 막히진 않음

## 7. **대반전**: UDP native 가 처음부터 IPsec 을 안 쓰고 있었음

검증 도중 P-CSCF 네임스페이스에서 직접 raw UDP 를 쏴보고 tcpdump 로 확인해봤다.

### 7.1 현재 코드 경로 (`SOCK_RAW + IP_HDRINCL`)

```
04:19:48.844216 IP 172.22.0.21.5100 > 10.20.20.2.6100: UDP, length 98
04:19:48.844271 IP 172.22.0.21.5100 > 10.20.20.2.6101: UDP, length 98
```

**plaintext UDP** 로 그대로 나감.  ESP 래핑 없음.

### 7.2 SOCK_DGRAM 로 보내봤더니

```
04:20:39.674358 IP 172.22.0.21 > 10.20.20.2: ESP(spi=0x00003ec1,seq=0x58f), length 500
04:20:39.711959 IP 10.20.20.2 > 172.22.0.21: ESP(spi=0x00001000,seq=0x2d3), length 756
04:20:39.712016 IP 172.22.0.21 > 10.20.20.2: ESP(spi=0x00003ec1,seq=0x590), length 84
```

커널이 xfrm OUT policy 를 보고 **ESP 로 잘 감싸서 보내고**, **A16 이 756 바이트 ESP 로 실제 응답**.

### 7.3 근본 원인

Linux 의 `SOCK_RAW + IPPROTO_UDP + IP_HDRINCL` 조합은 ip output 경로에서 **xfrm output 훅을 건너뛰고 직접 dev_queue_xmit 으로 내보낸다**.  `ipsec_native.py` 의 `_RAW_DRIVER_SCRIPT` 가 정확히 이 조합을 쓰고 있었음.

즉 **native 라는 이름을 달고 있는 UDP 경로가 실제로는 IPsec 을 적용하지 않은 plaintext UDP 였다**는 것.

A31 이 과거에 붙던 건 A31 의 IMS 스택이 **plaintext UDP 도 받아주는 관대한 구현** 이어서였을 뿐, 우리가 IPsec 을 쓰고 있었기 때문이 아니었다. A16/Pixel 처럼 spec-strict 한 단말에는 plaintext 가 드랍되어 모두 타임아웃.

이 사실을 안 상태에서 다시 보면:
- 기존 `bypass` 모드와 `native` 모드가 네트워크 레벨에서 사실상 구분이 없었음
- docs 와 코드의 "native IPsec" 이 misrepresentation 이었음
- A/B/C 복잡한 TCP plan 이 **아예 필요 없었다**: UDP native 만 제대로 고치면 A16 이 즉시 붙음

### 7.4 수정 (commit `cf3e87b`)

- `_RAW_DRIVER_SCRIPT` → `_UDP_DRIVER_SCRIPT` 로 리네임 및 내용 교체:
  ```python
  sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
  sock.setsockopt(SOL_SOCKET, SO_REUSEADDR, 1)
  try: sock.setsockopt(SOL_SOCKET, SO_REUSEPORT, 1)
  except: pass
  sock.bind((src_ip, src_port))
  sock.sendto(payload, (dst_ip, dst_port))
  ```
- `_RAW_SOCKET_PROBE_SCRIPT` → `_UDP_DGRAM_PROBE_SCRIPT` (SOCK_DGRAM 프로브로)
- 모듈 내부 reference 업데이트
- 주석에 **"왜 SOCK_RAW 를 썼던 이전 구현이 잘못이었는지"** 명시적 기록 — 이후 누군가 다시 raw 로 돌아가려 하지 않도록

744 테스트 pass. Push 는 보류 (서버에서 실제 A16 UDP native 캠페인으로 end-to-end 검증 후 진행 예정).

## 8. TCP native 의 운명

오늘 중간 커밋들 (`b6db2a6`, `720e286`) 로 넣은 TCP SOCK_STREAM 경로는 **A16 같은 single-stream TCP UE 에는 여전히 무용**하다.  단, 구조적으로 UE 가 새 TCP 연결을 받아주는 스택이라면 동작하도록 남겨두었음. 이번에 `cf3e87b` 이후에도 제거하지는 않았다. 이유:

- 코드 자체가 짧고 격리되어 있음
- UDP native 가 이제 제대로 작동하므로 A16 기준으로는 TCP 가 필수가 아님
- 다른 UE/스택 대상 테스트 시 재사용 가능성 있음
- 필요할 때 Approach B (TCP_REPAIR) 나 C (kamailio 프록시) 로 확장 가능

## 9. 배운 것 / 향후 확인 필요

### 9.1 배운 것

1. **docs 의 "자동 resolve" 는 경계를 명확히 해야 한다**: host resolve 와 port resolve 가 서로 다른 수준으로 자동화되어 있었는데, CLAUDE.md / README 에는 통합된 것처럼 서술되어 있었음. 오늘 port 쪽도 실제로 live resolve 되도록 맞췄음.
2. **"native IPsec" 은 이름값을 해야 한다**: 기능 이름이 네트워크 레벨 실제 동작을 오해하게 만들면 디버깅이 엉뚱한 방향으로 간다. 오늘 `SOCK_RAW + IP_HDRINCL` 이 xfrm 을 건너뛰는 사실은 kernel 문서로 추적 가능하지만, 이 프로젝트의 commit 히스토리엔 그 전제가 명시된 적이 없어 보였다.
3. **호스트 환경 (CPU governor, USB 버스) 의 재부팅 이후 복구 루틴이 빠져있었다**: `setup_host.sh --all` 을 다시 한 번 돌려야 하는 요소였는데, 자동 로딩 단계에 governor 가 없었음. 오늘 넣었다.
4. **A31 대응 "성공" 이 false positive 였을 가능성**: 과거 A31 ring-through 는 IPsec 때문이 아니라 A31 이 plaintext 도 받아주는 스택이라서 가능했던 것. 이는 `docs/A31_REAL_UE_GUIDE.md` 에서 재검토가 필요할 수 있다.

### 9.2 확인 필요 / TODO

- **A16 UDP native end-to-end**: 서버에 `cf3e87b` 동기화 후 실제 MESSAGE 캠페인이 normal/suspicious 판정을 받는지. 지금까지는 tcpdump 레벨에서 ESP 왕복만 확인된 상태.
- **port_pc vs port_ps for UDP**: 현재 UDP 는 `port_pc` 타겟 유지. A16 의 경우 실제로 어느 쪽이 listener 를 열고 있는지 확인해서, 필요하면 TCP 와 같이 port_ps 로 전환.
- **A31 회귀 테스트**: UDP native 가 ESP 로 바뀐 이후 A31 이 여전히 정상 응답하는지. A31 이 "ESP 전용 listener + plaintext listener" 둘 다 갖고 있다면 regression 없음. plaintext 전용만 있었다면 regression.

- **TCP 경로**: 현재는 UE 가 TCP listener 를 여는 경우에만 동작. A16 처럼 single-stream 스택 대상으로 확장하려면 Approach C (kamailio 프록시) 가 현실적 루트.
