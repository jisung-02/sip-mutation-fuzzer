# 장기 퍼징 결과 시스템 설계

> 2026-04-13 작성. 장기 캠페인(수백~수천 케이스)을 위한 오라클 기본 활성화,
> 실시간 대시보드, 케이스별 증거 수집, HTML 리포트 생성 설계 문서.

---

## 1. 현황 및 문제

### 1.1 현재 오라클 기본값

| 구성 요소 | 기본값 | 문제 |
|-----------|--------|------|
| SocketOracle | 항상 ON | 문제 없음 |
| ProcessOracle | `check_process=True`, `process_name="baresip"` | A31 실기기에 baresip 없음 → crash 오탐 |
| ADB | `adb_enabled=False` | 실기기인데 꺼져있음 → 단말 크래시 놓침 |
| Pcap | `pcap_enabled=False` | 증거 없이 verdict만 남음 |

### 1.2 현재 결과물

- **campaign.jsonl**: 한 줄씩 JSON — 사람이 읽기 어려움
- **pcap/case_XXXXXX.pcap**: `--pcap` 플래그 수동 활성화 필요
- **adb_snapshots/**: crash/stack_failure 시에만 저장, `--adb` 수동 활성화 필요
- 통계는 캠페인 종료 후 `campaign report` 명령으로 JSON 출력 — 시각화 없음

### 1.3 CaseGenerator 한계

현재 `CaseGenerator`는 method × layer × strategy **고유 조합**만 생성.
`--max-cases 1000` 지정해도 조합 수(예: 1 method × 2 layer × 2 strategy = 4개)만큼만 나옴.
→ seed 기반 반복 지원 필요 (별도 이슈, 이 문서에서는 다루지 않음).

## 2. 설계: 4가지 개선

### 2.1 실기기 오라클 기본 활성화

**목표**: `--mode real-ue-direct`일 때 사용자가 일일이 플래그를 켜지 않아도
실기기에 맞는 오라클 구성이 자동 적용.

**변경 사항** (`campaign/contracts.py` — `CampaignConfig` model_validator):

```
mode == "real-ue-direct" 감지 시:
  check_process = False          (baresip 체크 불필요)
  adb_enabled = True             (단말 logcat 모니터링)
  pcap_enabled = True            (증거 보존)
  pcap_interface = "br-volte"    (Docker bridge에서 캡처)
```

- 사용자가 명시적으로 `--no-adb`, `--no-pcap` 지정하면 그 값 존중
- softphone 모드는 기존 기본값 유지 (변경 없음)

**CLI 변경** (`campaign/cli.py`):
- `adb`, `pcap` 옵션의 기본값을 `None`으로 변경 (3-state: True/False/None)
- `None`이면 mode에 따라 자동 결정, 명시적 True/False면 사용자 지정 존중

### 2.2 실시간 콘솔 대시보드

**목표**: 장기 퍼징 중 터미널에서 진행 상황을 한눈에 파악.

**출력 형태** (stderr, 10케이스마다 갱신):

```
══════════════════════════════════════════════════════
 Campaign: a3f8c2  |  847/1000  |  00:42:15  |  0.33/s
 
 normal 612 (72%)  suspicious 31 (4%)  timeout 198 (23%)
 crash 0  stack_failure 6 (1%)  unknown 0  infra 0
 
 ADB: healthy  |  Pcap: ON (br-volte)
 Last: [#847] INVITE wire/safe seed=846 → timeout (5.0s)
══════════════════════════════════════════════════════
```

**구현 위치**: `campaign/core.py` — `CampaignExecutor.run()` 루프 내부

**설계**:
- 신규 클래스 `ConsoleProgressReporter`
  ```python
  class ConsoleProgressReporter:
      def __init__(self, total: int, campaign_id: str) -> None: ...
      def update(self, case_result: CaseResult, summary: CampaignSummary) -> None: ...
      def finalize(self) -> None: ...
  ```
- `update()` 호출 시:
  - 경과 시간 계산 (start_time 기준)
  - 처리 속도 (cases/sec)
  - verdict별 카운트 + 백분율
  - ADB collector 건강 상태
  - 직전 케이스 한 줄 요약
- 매 케이스마다 마지막 줄(Last:)만 갱신, 매 N케이스(기본 10)마다 전체 블록 출력
- `--quiet` 옵션 추가 시 기존 한 줄 출력만 유지

**기존 출력과의 관계**:
- 현재 `print(f"{label} → {case_result.verdict}...")` 출력은 대시보드 내 Last 줄로 통합
- crash/stack_failure 경고는 대시보드 아래에 별도 출력 (기존처럼)

### 2.3 주목 케이스 증거 수집 (Interesting Case Collector)

**목표**: crash, stack_failure, suspicious verdict가 나온 케이스의 증거를
개별 디렉토리에 자동 수집. 사후 분석과 HTML 리포트 연동에 사용.

**디렉토리 구조**:
```
results/a31/
├── campaign.jsonl
├── pcap/
│   ├── case_000031.pcap
│   └── case_000847.pcap
└── interesting/
    └── case_000031/
        ├── summary.json       # verdict, strategy, seed, mutation_ops, reason
        ├── sent.sip           # 보낸 SIP 패킷 원문 (wire_text)
        ├── response.sip       # 받은 SIP 응답 원문 (있으면)
        ├── capture.pcap       # 해당 케이스 pcap (복사 또는 심볼릭링크)
        ├── adb_log.txt        # 해당 시점 ADB logcat 스냅샷
        └── mutation_ops.txt   # 적용된 변이 연산 목록
```

**구현**:

신규 모듈 `campaign/evidence.py`:
```python
class EvidenceCollector:
    """주목할 verdict의 케이스별 증거를 수집한다."""

    INTERESTING_VERDICTS: frozenset[str] = frozenset({
        "crash", "stack_failure", "suspicious"
    })

    def __init__(self, base_dir: Path, enabled: bool = True) -> None:
        self._base_dir = base_dir / "interesting"
        self._enabled = enabled

    def should_collect(self, verdict: str) -> bool:
        return self._enabled and verdict in self.INTERESTING_VERDICTS

    def collect(
        self,
        case_result: CaseResult,
        *,
        sent_payload: str | bytes | None = None,
        pcap_path: str | None = None,
        adb_snapshot: str | None = None,
    ) -> str | None:
        """증거를 수집하고 디렉토리 경로를 반환한다."""
        ...
```

**collect() 내부 동작**:
1. `interesting/case_{case_id:06d}/` 디렉토리 생성
2. `summary.json` 작성: case_result에서 verdict, reason, method, layer, strategy, seed, mutation_ops, elapsed_ms, response_code, reproduction_cmd 추출
3. `sent.sip` 작성: 변이 적용된 최종 wire_text (또는 hex dump for byte layer)
4. `response.sip` 작성: raw_response가 있으면 저장
5. `capture.pcap`: pcap_path가 있으면 해당 파일을 이 디렉토리로 복사
6. `adb_log.txt`: ADB 스냅샷이 있으면 저장
7. `mutation_ops.txt`: 변이 연산 목록 (한 줄에 하나)

**CampaignExecutor 연동**:
- `_execute_case()` 및 `_execute_mt_template_case()`에서 verdict 판정 후
  `EvidenceCollector.collect()` 호출
- sent_payload는 SendArtifact 생성 시점의 wire_text/packet_bytes를 캡처하여 전달

**pcap 양방향 캡처**:
- 현재 tcpdump 필터: `"udp port 5060 or tcp port 5060"` → 송신+수신 모두 캡처됨 (변경 불필요)
- br-volte 인터페이스에서 캡처하면 pcscf 컨테이너의 나가는 패킷과 UE에서 돌아오는 응답 패킷 모두 포함

## 4. 데이터 흐름

```
[CampaignExecutor.run() 루프]
    │
    ├─ 매 케이스 실행
    │   ├─ _execute_case() / _execute_mt_template_case()
    │   │   ├─ 패킷 생성 + 변이
    │   │   ├─ pcap 시작
    │   │   ├─ 송신
    │   │   ├─ pcap 종료
    │   │   ├─ oracle 판정
    │   │   └─ CaseResult 반환 (sent_payload 포함)
    │   │
    │   ├─ ResultStore.append(case_result)          → campaign.jsonl
    │   ├─ EvidenceCollector.collect(case_result)    → interesting/case_XXXXXX/
    │   └─ ConsoleProgressReporter.update()          → stderr 대시보드
    │
    └─ 루프 종료
        ├─ ResultStore.write_footer()
        ├─ ConsoleProgressReporter.finalize()        → 최종 요약 출력
        └─ HtmlReportGenerator.generate()            → report.html
```

## 6. 제약 및 고려사항

### 6.1 pcap 양방향 확인
현재 tcpdump 필터 `"udp port 5060 or tcp port 5060"`는 **나가는 패킷 + 들어오는 응답 모두 캡처**.
br-volte 인터페이스 기준, pcscf 컨테이너에서 나가는 SIP와 UE에서 돌아오는 100/180/200이
모두 하나의 pcap에 포함됨. 변경 불필요.

### 6.2 HTML 리포트 크기
- 1000 케이스 기준: 전체 테이블 ~200KB, interesting 케이스 인라인 시 ~500KB 이내
- pcap은 인라인하지 않음 (다운로드 링크만)
- 10,000 케이스 이상: 테이블 페이지네이션 또는 interesting만 표시하는 간소화 모드 고려

### 6.3 성능 영향
- EvidenceCollector: 파일 I/O는 interesting verdict에만 발생 → 대부분 케이스에서 overhead 없음
- ConsoleProgressReporter: stderr 출력만, 매 케이스 1 print → negligible
- HtmlReportGenerator: 캠페인 종료 후 1회 실행 → 런타임 영향 없음

### 6.4 기존 호환성
- softphone 모드: 모든 기본값 유지, 변경 없음
- `--no-adb --no-pcap` 명시 시: real-ue-direct에서도 기존처럼 최소 구성 가능
- jsonl 포맷: `sent_payload` 필드 추가만 — 기존 파서는 무시하면 됨

### 6.5 kamailio 로그 (현재 불필요)
- `LogOracle`은 `--log-path` 지정 시에만 활성화
- real-ue-direct 기본값에서 log_path는 None 유지
- 추후 kamailio 로그 모니터링이 필요해지면 `--log-path pcscf` (docker mode) 지정
