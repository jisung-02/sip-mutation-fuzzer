# 작업 로그 — 2026-04-11

## 목표

A31 real-ue-direct MT INVITE 템플릿 퍼징 통합 — 자동 캠페인(`fuzzer campaign run
--mode real-ue-direct`)이 **실제로 Samsung A31을 울리게** 만들고, 그 위에
wire/byte 변이를 얹을 수 있는 파이프라인 구축.

phase-1 완료 기준: `--strategy identity` 케이스가 `/tmp/send_mt_invite.py`와
동일한 시그널 형태로 A31에 MT INVITE를 전달해 단말이 실제로 울린다.

## 1. 설계 / 플랜 수립

### 근거 문서
- `docs/이슈/A31-real-ue-direct-INVITE-성공-기록.md` — 2026-04-11 수동 검증으로
  A31이 실제 울린 5가지 조건 기록
- `memory/project_a31_real_ue_direct_breakthrough.md` — 재사용 가능한 핵심 규칙

### 5가지 성공 조건 (수동 검증)
1. Source IP = **172.22.0.21** (등록 P-CSCF) — A31 IMS 앱 레이어 화이트리스트
2. 3GPP MT-INVITE 완전 포맷 — Record-Route ×3, P-* 헤더 전체, +sip.instance/
   +g.3gpp.* Contact feature tags, AMR/AMR-WB SDP + curr/des:qos preconditions
3. Request-URI = `sip:<IMPI>@10.20.20.8:<port_pc>;alias=10.20.20.8~<port_ps>~1`
   (port_pc/port_ps는 재등록마다 rotate)
4. 평문 UDP 충분 — ESP 불필요
5. IP 단편화는 over-the-air 경로에서만 치명적, Docker br-volte 내부망은 안전

### 설계 핵심
- **템플릿은 wire-text로 보관** — packet catalog 의존 없이 `{{slot}}` 치환
- **Mutator 진입점 신설**: `SIPMutator.mutate_editable(EditableSIPMessage, MutationConfig)`
  → `MutatedWireCase` (기존 `MutatedCase`는 `original_packet` 필수라 우회 못 함)
- **Sender 신규 경로**: `TargetEndpoint.bind_container` → `docker exec -i <ctr>
  python3 -c <driver>`로 컨테이너 netns에서 송신
- **`identity` 전략** — zero-op baseline, 오라클과 바이트 동일 송신
- **Via/Contact preserve 플래그** — template Via를 authoritative로 취급

## 3. 원격 E2E 검증 — 삽질과 해결

### 3-1. port_pc 조회 실패

```
[ERROR] mt-template executor error: could not resolve protected ports for UE
        via container 'pcscf'. Ensure the UE is registered and logs are available.
```

**원인**: `resolve_ue_protected_ports`의 xfrm fallback 파서가
`if stripped.startswith("sport"):` 조건을 썼는데, 실제 xfrm state 출력에서
sport/dport가 포함된 줄은 `sel src .../32 dst .../32 sport N dport M`로
**`sel`로 시작**. 매칭이 안 돼 fallback도 실패했다.

**수정**: `sel`로 시작하고 sport/dport를 포함하는 줄을 매칭하도록 변경. 동시에
SA 헤더 라인(`src X dst Y`)으로 방향을 추적해 A31(10.20.20.x)이 소스인 SA의
sport만 수집.

### 3-2. port_pc / port_ps 혼동

수정 후 `(8101, 8102)` 반환 — 잘못됨. 현재 xfrm state는:
```
sel src 10.20.20.8/32 dst 172.22.0.21/32 sport 8100 dport 5103
sel src 10.20.20.8/32 dst 172.22.0.21/32 sport 8101 dport 6103
```

A31→pcscf SA가 2개고 각각 sport가 8100(port_pc), 8101(port_ps). 내 파서는
마지막 sport만 집어서 port_ps를 port_pc로 착각.

**수정**: A31 sport를 전부 수집 후 `min()`을 port_pc로, `port_pc+1`을 port_ps로.
관례상 port_ps = port_pc + 1.

### 3-3. Fragmentation guard

```
[ERROR] template payload exceeds one-fragment UDP safety threshold (1400 bytes)
```

렌더된 MT INVITE는 2460 bytes. 1400 바이트 guard에 걸림.

**원인 재해석**: 성공 기록에 "IP fragmentation은 over-the-air LTE 경로에서만
치명적"이라고 명시돼 있었음. `bind_container` 경로는 Docker br-volte 내부망이라
fragmentation이 안전.

**수정**: `campaign/core.py`의 frag guard에 `and self._target.bind_container is None`
조건 추가. host-direct UDP 경로만 guard 적용, container 경로는 bypass.

### 3-4. 엉뚱한 `crash` verdict

```
[1/1] INVITE wire/identity seed=0 → crash ( 101ms)
  [CRASH] reason: process 'baresip' not found after send
```

A31 크래시가 아니라, oracle이 `process_name="baresip"` 기본값으로 서버에 없는
baresip 프로세스를 찾다가 crash verdict를 내고 있었음.

**수정**: `--no-process-check` 플래그 추가해서 oracle process check 비활성화.

### 3-5. 목적지 포트 잘못 — port_ps vs port_pc

ESP/xfrm policy 분석(`sport 5103 dport 8101`)으로 이어지는 rabbit hole에서
"port_ps로 보내야 한다"고 오판하고 `mt_target.port = port_ps`로 수정했음.

**근거 확인**: `/tmp/send_mt_invite.py`의 실제 코드:
```python
sock.sendto(payload, (UE_IP, UE_PORT_PC))   # ← port_pc 로 송신!
```

메모리 노트도 명시: "Plaintext UDP works — ESP/IPsec is not required. You can
use any source port." — ESP policy 매칭은 불필요.

**수정**: `mt_target.port = port_pc`로 되돌림. 주석에 원본 스크립트 근거 기록.

### 3-6. 바이트 단위 비교

`/tmp/send_mt_invite.py`의 `build(local_port)`와 내 `render_mt_invite(slots)` 출력을
바이트 단위 diff:

- orig: 2455 bytes, mine: 2460 bytes
- 60 lines 모두 동일한 순서/위치
- Content-Length 776 = 776 (일치)
- 차이는 randomize된 식별자 4개: `branch`, `ftag`, `Call-ID`, `icid` 값뿐

→ 구조적/시맨틱으로 원본과 동치. A31 거부 원인 아님.

## 5. 남은 이슈 — Via sent-by 포트 불일치

```
Via: SIP/2.0/UDP 172.22.0.21:5100;branch=...
```

- 템플릿은 `local_port=5100`으로 하드코딩
- `_DRIVER_SCRIPT`는 `sock.bind((bind_host, 0))`로 랜덤 포트 바인딩
- A31은 Via의 sent-by 포트(5100)로 100 Trying / 180 Ringing 응답 송신
- 내 랜덤 포트 소켓은 그걸 수신 못 함 → recvfrom timeout → fuzzer verdict `timeout`

**패킷은 정상 전달, 벨도 울림, 응답만 수신 실패.**

### 수정 방향 (phase-1.5)
1. `send_via_container`에 `bind_port: int = 0` 파라미터 추가
2. `_DRIVER_SCRIPT`를 `sock.bind((bind_host, bind_port))`로 수정,
   0이면 현행 random 유지
3. `_execute_mt_template_case`에서 free high port 하나 선택해 `slots.local_port`와
   `send_via_container(bind_port=...)` 양쪽에 전달
4. 완료 시 100 Trying / 180 Ringing이 `SocketObservation`으로 수집돼 oracle이
   `normal`/`provisional` verdict를 내게 됨

## 12. Phase-2 (후속)

phase-1이 완전히 동작하므로 다음 단계로 wire/byte 변이 퍼징 가능:

```bash
uv run fuzzer campaign run \
  --mode real-ue-direct --transport UDP \
  --target-host 10.20.20.8 --target-msisdn 111111 \
  --impi 001010000123511 \
  --methods INVITE --layer wire,byte --strategy identity,default \
  --mt-invite-template a31 --bind-container pcscf \
  --preserve-contact --preserve-via --mt-local-port 15100 \
  --max-cases 10 --timeout 10 --no-process-check \
  --pcap --pcap-interface br-volte --pcap-dir results/a31/pcap \
  --adb --adb-serial <A31 serial>
```

기대 동작:
- case_000000 (identity) = 항상 A31 벨 울림 + `normal (180)` baseline
- case_000001~ = wire/byte 변이 적용, verdict `normal`/`suspicious`/`timeout` 혼재
- `crash`/`stack_failure` 발생 시 adb snapshot 자동 덤프
- `br-volte` interface pcap으로 컨테이너 송수신 플로우 전체 기록
