## 1. UE 서브넷 라우트 추가 (`poe add-ue-routes`)

### 문제
Ubuntu 서버에서 UE IMS 서브넷(10.20.20.0/24, 10.10.10.0/24)으로 가는 라우트가 없었음.  
패킷이 UPF(172.22.0.8)가 아닌 공인 게이트웨이(163.180.185.1)로 나가서 UE 도달 불가.

### 원인 파악
UPF 컨테이너 내부의 ogstun/ogstun2 인터페이스가 UE 서브넷을 처리하지만,  
Ubuntu 호스트에는 해당 서브넷 라우트가 없어 `via 163.180.185.1 dev eno1`으로 잘못 나감.

### 해결

**`scripts/add_ue_routes.py`** 신규 생성  
- `sudo -v`로 sudo 자격증명 캐시 (한 번만 비번 요청)
- `sudo -n ip route replace 10.10.10.0/24 via 172.22.0.8`
- `sudo -n ip route replace 10.20.20.0/24 via 172.22.0.8`
- Python subprocess 사용 이유: poe `shell` 타입은 TTY에서 stdin을 분리하므로 `sudo` 비번 프롬프트 불가

**`pyproject.toml`** 에 `poe add-ue-routes` 태스크 추가
```toml
[tool.poe.tasks.add-ue-routes]
help = "Add host routes to UE subnets (10.10.10.0/24, 10.20.20.0/24) via UPF"
cmd = "python scripts/add_ue_routes.py"
```

### 부수 버그 수정 — TOML 스코프 문제
`epc-run`, `epc-stop`, `epc-status`, `epc-logs`가 `[tool.poe.tasks.enb-run]` 서브테이블 아래에  
잘못 위치해 poe가 "Unrecognized option 'epc-run'" 에러 발생.  
→ `[tool.poe.tasks]` 섹션 내부(서브테이블 헤더 이전)로 이동하여 수정. (커밋 `1cd1ca7`)

### 버그 수정 1 — `_parse_kamctl_output()` 다중 UE 처리 오류

**파일**: `src/volte_mutation_fuzzer/sender/real_ue.py`

**문제**: UE가 여러 대 등록된 경우(`kamctl ul show` 전체 출력 사용 시),  
`direct_match`가 AOR 필터링 없이 출력에서 첫 번째로 발견된 Contact를 반환.  
→ UE2(222222)가 먼저 나오면 UE1(111111) 조회 시 잘못된 IP 반환.

**수정**: AOR 섹션(`AOR:` 헤더) 유무로 분기
- AOR 섹션 없음 (단일 UE 응답) → `direct_match` 사용 (안전)
- AOR 섹션 있음 (다중 UE 응답) → MSISDN 필터링 후 AOR-based 검색

```python
# Before (buggy)
direct_match = _KAMCTL_CONTACT_PATTERN.search(output)
if direct_match is not None:
    return UEContact(...)  # 잘못된 UE 반환 가능

# After
has_aor_sections = any(line.startswith("AOR:") for line in lines)
if not has_aor_sections:
    direct_match = _KAMCTL_CONTACT_PATTERN.search(output)
    ...
else:
    # MSISDN 기반 필터링
    current_aor_matches = False
    for line in lines:
        if line.startswith("AOR:"):
            current_aor_matches = msisdn in line
        ...
```

## 10. Samsung A31 IPsec/IMS APN 동작 분석

### Samsung A31이 internet APN을 쓰는 이유
1. EPC 재시작 직후 P-CSCF가 완전히 준비되기 전에 IMS APN PDN 수립 시도
2. IPsec 협상 실패 또는 IMS 등록 실패
3. Samsung IMS 스택이 "IMS 불가" 상태로 마킹하고 retry 포기 → internet APN 폴백
4. 이 상태를 **캐시**해서 자동으로 다시 시도하지 않음

### APN 수동 토글이 동작하는 이유
- 설정에서 internet APN → IMS APN으로 수동 전환 시 Samsung IMS 서비스 매니저가 **상태 리셋**
- IMS 등록 처음부터 다시 시도 → 이 시점엔 IMS 인프라 안정 → 등록 성공
- 재등록 후 `10.20.20.x:31800` IMS APN IP로 P-CSCF에 등록됨

### Huawei와의 차이
Huawei는 IMS 클라이언트가 retry를 더 공격적으로 수행해 결국 IMS APN 수립 성공.  
Samsung은 첫 실패 후 캐싱으로 인해 수동 개입 필요.

## 18. INVITE 전송 분석 — Via `received` 파라미터 문제

### 시도 및 관찰

INVITE를 `10.20.20.10:5060 TCP`로 전송했을 때 100 Trying은 오지만 이후 응답 없음. 단말 화면에 아무것도 안 뜸.

tcpdump (ogstun2) 분석:

```
172.22.0.1 → 10.20.20.15:5060  INVITE
  Via: SIP/2.0/TCP 172.22.0.21:5060;branch=...;rport;received=172.22.0.1
10.20.20.15 → 172.22.0.1       100 Trying
(이후 응답 없음)
```

### 근본 원인

현재 퍼저는 Via에 `rport` 파라미터를 강제로 추가함. RFC 3581에 따라 A31이 Via 응답에 `rport=PORT`와 `received=실제발신지IP`를 채움.

- Via host = `172.22.0.21` (P-CSCF)
- received = `172.22.0.1` (퍼저 서버)
- **두 값이 달라서 Samsung IMS 스택이 "신뢰할 수 없는 발신지"로 판단, 처리 거부**

### 캡스톤과의 차이

| 항목 | 캡스톤 dumpipe | 현재 프로젝트 |
|------|---------------|--------------|
| 전송 방식 | UDP | TCP |
| Via host | 퍼저 실제 IP (172.22.0.1) | P-CSCF IP (172.22.0.21) |
| rport | 없음 (기존 값 유지) | **강제 True** |
| received 파라미터 | 없음 (Via host == 실제 발신지) | `received=172.22.0.1` 추가됨 |
| Samsung 반응 | 신뢰 → 처리 | 신뢰 거부 → 100 Trying만 |

## 21. UDP 소켓 수신 버그 수정 (connect() → sendto())

### 문제

`real-ue-direct` UDP 경로에서 `sock.connect((host, port))` + `sock.send()` 사용 중이었음.
Linux에서 connected UDP 소켓은 연결된 peer(host:port)에서 오는 패킷만 수신.
A31은 수신 7900 (port_pc), 송신 7901 (port_ps) — 다른 포트.
→ A31 응답이 7901에서 오면 커널이 매칭 실패 → **ICMP port unreachable** 전송.

tcpdump ogstun2에서 확인:
```
172.22.0.1.55746 > 10.20.20.8.7900: UDP, length 1647   ← INVITE 전달됨
10.20.20.8.7901 > 172.22.0.1.55746: UDP, length 325    ← A31이 응답 (70ms)
172.22.0.1 > 10.20.20.8: ICMP udp port 55746 unreachable  ← 소켓 없음
```
A31이 응답을 3회 재전송 후 포기. fuzzer는 timeout.

### 수정 (`src/volte_mutation_fuzzer/sender/core.py`)

로컬 IP 확인용으로만 probe 소켓 connect() 후 즉시 닫고,
실제 송수신은 unconnected 소켓 + bind() + sendto() 사용:

```python
# Before
with socket.socket(AF_INET, SOCK_DGRAM) as sock:
    sock.connect((resolved.host, resolved.port))
    local_host, local_port = sock.getsockname()
    sock.send(payload)
    observations = self._read_udp_observations(sock, ...)

# After
with socket.socket(AF_INET, SOCK_DGRAM) as _probe:
    _probe.connect((resolved.host, resolved.port))
    local_host, _ = _probe.getsockname()
# unconnected 소켓: 어떤 포트에서 오는 응답도 recvfrom()으로 수신 가능
with socket.socket(AF_INET, SOCK_DGRAM) as sock:
    sock.bind((local_host, 0))
    _, local_port = sock.getsockname()
    sock.sendto(payload, (resolved.host, resolved.port))
    observations = self._read_udp_observations(sock, ...)
```

### 결과

수정 후 A31 응답 수신 성공 → `400 Parsing Failed` (SIP 내용 문제로 이어짐)

## 23. Samsung IMS 앱레이어 소스 IP 검증 문제

### 현상

TCP로 전송 시 `100 Trying` 수신, 이후 응답 없음. A31 화면에 착신 표시 없음.

### P-CSCF 경유 정상 MT INVITE 분석

222222 → 111111 실제 전화 시 P-CSCF 로그:
```
Request URI: sip:001010000123511@10.20.20.8:7900  (IMSI 기반)
ipsec_forward() → sip:10.20.20.8:7900             (ESP 암호화)
Source: 172.22.0.21 (P-CSCF IP)
→ 180 Ringing ✅
```

fuzzer 직접 전송:
```
Source: 172.22.0.1 (fuzzer IP)
→ 100 Trying → 이후 침묵 ❌
```

### 결론

Samsung IMS 앱레이어가 **소스 IP를 기반으로 신뢰 검사** 수행:
- 172.22.0.21 (P-CSCF) → 신뢰 → 180 Ringing
- 172.22.0.1 (fuzzer) → 거부 → silent drop

XFRM 레벨이 아닌 Samsung IMS 애플리케이션 레이어에서 필터링.
(100 Trying은 SIP 트랜잭션 레이어에서 자동 처리, 앱레이어는 별개)

### 해결 방안 (미결)

**iptables SNAT 방식** — fuzzer 패킷이 P-CSCF IP처럼 보이게:
```bash
# 송신: fuzzer → A31을 P-CSCF IP로 위장
iptables -t nat -A POSTROUTING -s 172.22.0.1 -d 10.20.20.8 -j SNAT --to-source 172.22.0.21

# 수신: A31 → P-CSCF 응답을 fuzzer로 리다이렉트
iptables -t nat -A PREROUTING -s 10.20.20.8 -d 172.22.0.21 -p tcp -j DNAT --to-destination 172.22.0.1
```
→ A31 입장에서 P-CSCF와 통신하는 것처럼 보임, fuzzer가 응답 수신 가능.
→ 실제 P-CSCF 트래픽과 포트 충돌 주의, 포트 필터 추가 필요.

## 24. iptables SNAT 시도 및 실패

### 시도

```bash
sudo iptables -t nat -A POSTROUTING -s 172.22.0.1 -d 10.20.20.8 -p tcp --dport 7900 -j SNAT --to-source 172.22.0.21
```

### 실패 원인

SNAT 적용 후 TCP 연결 자체가 timeout (bytes_sent: 0).

**원인 분석**: SYN-ACK 응답 경로 문제.
- fuzzer → SNAT → (172.22.0.21) → GTP → A31
- A31이 SYN-ACK을 172.22.0.21(P-CSCF)로 전송
- UPF 컨테이너가 br-volte를 통해 P-CSCF 컨테이너로 직접 라우팅
- 호스트 커널 conntrack PREROUTING을 거치지 않음 → P-CSCF가 RST 전송
- Docker 브리지 네트워크 내 컨테이너 간 직접 라우팅으로 인해 conntrack un-NAT 불가

### 결론

Docker 네트워크 토폴로지에서 SNAT으로 TCP 소스 스푸핑 불가.
SNAT 룰 제거 후 원상 복구.

### 유력한 다음 방향

- P-CSCF가 IPsec 적용하여 A31에 전달 (소스 IP = 172.22.0.21)
