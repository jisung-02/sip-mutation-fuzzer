⏺ VolteMutationFuzzer 파이프라인 개요

  실기기(UE) 대상 VoLTE/IMS SIP 프로토콜 취약점 탐색 도구입니다. 큰 그림은
  다음과 같습니다.

  ---
  전체 흐름

  CLI 입력
    ↓
  CampaignConfig (설정 + MSISDN→IP 자동 resolve)
    ↓
  CaseGenerator (method × profile × layer × strategy 조합 생성)
    ↓
  Generator (SIP 패킷 생성)
    ↓
  Mutator (변이 적용)
    ↓
  Sender (실기기로 UDP 전송)
    ↓
  Oracle (응답 판정: normal / suspicious / timeout / crash)
    ↓
  ResultStore + PcapCapture + ADB/iOS 스냅샷

  ---
  핵심 컴포넌트 역할

  1. Generator — SIP 패킷 원본 생성
  - real-ue-direct 모드에서는 실제 캡처 기반 MT-INVITE 템플릿
  (mt_invite_a31.sip.tmpl)을 사용
  - 18개 슬롯(IMPI, port_pc/port_ps, Call-ID 등)을 동적으로 채워 렌더링

  2. Mutator — 변이 엔진, 두 축으로 제어
  - layer: model (SIP 구조 인식) → wire (텍스트 레벨) → byte (바이트 레벨)
  - profile: legacy / delivery_preserving / ims_specific / parser_breaker
  - strategy=default는 요청값일 뿐, 실제 concrete strategy는 profile + layer + 
  seed로 결정됨

  3. Sender (real-ue-direct 경로)
  - resolve_protected_ports() → UE의 port_pc/port_ps 실시간 조회
  - ipsec_mode=null: host에서 직접 소스 IP를 P-CSCF IP(172.22.0.21)로 스푸핑해
  UDP 전송
  - ipsec_mode=bypass: Docker P-CSCF 컨테이너 netns 진입 후 전송

  4. Oracle — SIP 응답 → verdict 분류
  - 180/200 → normal, 4xx/5xx → suspicious, 무응답 → timeout, 프로세스 종료 →
  crash, logcat/iOS syslog 이상 → stack_failure

  5. Storage — JSONL 형식 결과 저장 + pcap + ADB/iOS 스냅샷

  ---
  주요 설계 포인트

  - mode(실행 경로)와 profile(변이 정책)은 독립 축 — 혼동 금지
  - 현재 실기기 baseline이 확립된 메서드는 INVITE뿐 (real_ue_baseline)
  - MSISDN→IP는 hardcoded 없이 항상 live resolver (kamctl → P-CSCF 로그 → xfrm
  state) 순으로 조회
  - 단편화 방지를 위해 payload 1400B 초과 시 경고 (fragment guard)

  4. Mutator — 변이 적용

  mutator.mutate_editable(editable, MutationConfig(
      seed=spec.seed,
      profile=spec.profile,   # legacy / delivery_preserving / ims_specific / 
  parser_breaker

  → MutatedWireCase (wire_text 또는 packet_bytes)

  변이 레이어 구조:
  EditableSIPMessage (wire text 파싱 결과)
      ↓ wire layer: 헤더/공백/CRLF/Content-Length 텍스트 조작
      ↓ byte layer: 바이트 직접 조작
  → 변이된 SIP 패킷

  identity strategy면 원본 그대로 통과 (baseline 확인용).

  ---
  5. Sender — 실기기로 UDP 전송

  ipsec_mode=null 경로:
  # 호스트에서 소스 IP를 P-CSCF IP(172.22.0.21)로 스푸핑
  socket.bind(("0.0.0.0", mt_local_port))  # Via sent-by 포트와 일치시킴
  socket.sendto(packet_bytes, (ue_ip, port_pc))

  ipsec_mode=bypass 경로:
  # Docker P-CSCF 컨테이너 netns 안에서 실행
  docker exec pcscf python3 send_udp.py ...

  INVITE는 응답을 전부 수집 (collect_all_responses=True). 이후 CANCEL teardown도
   자동으로 수행 (180 Ringing 수신 확인 후).

  ---
  6. Oracle — 응답 판정

  # 우선순위 순서로 판정
  if process_crashed:    → "crash"
  if stack_trace_found:  → "stack_failure"  (ADB logcat / iOS syslog)
  if timeout_exceeded:   → "timeout"
  if 4xx/5xx response:   → "suspicious"
  if 1xx/2xx response:   → "normal"

  Circuit Breaker: timeout/unknown이 연속으로 일정 횟수 이상 쌓이면 IPsec SA
  만료 여부를 체크하고 캠페인 중단.

  ---
  7. ResultStore + PcapCapture + ADB/iOS 스냅샷

  - ResultStore: 케이스마다 JSONL에 verdict, raw_request, raw_response,
  reproduction_cmd, mutation_ops 저장
  - PcapCapture: tcpdump로 케이스별 .pcap 파일 저장 (host <ue_ip> 필터)
  - ADB 스냅샷: suspicious/crash/stack_failure verdict면 logcat + screenshot
  저장
  - iOS 스냅샷: syslog + .ips 크래시 리포트 저장
  - 캠페인 종료 시 HTML 리포트 자동 생성

  ---
  1단계: CLI 입력 → CampaignConfig

  사용자가 입력한 플래그들이 CampaignConfig(Pydantic 모델)로 파싱·검증된다.

  --mode real-ue-direct
  --target-msisdn 111111
  --methods INVITE
  --profile ims_specific
  --layer wire
  --strategy default
  --max-cases 50

  이 시점에 MSISDN → UE IP resolve가 시작된다:

  RealUEDirectResolver.resolve() 호출
    1순위: kamctl ul show  (Kamailio DB에서 Contact IP 조회)
    2순위: P-CSCF kamailio 로그에서 "PCSCF: REGISTER" 라인 파싱
    3순위: ip xfrm state (IPsec SA selector에서 UE IP 추출)
    환경변수 override: VMF_MSISDN_TO_IP_111111=10.20.20.x
    실패 시: 명확한 에러 + raise (hardcoded fallback 없음)

  TargetEndpoint 객체도 이 시점에 생성된다. ipsec_mode=null이면
  bind_port=mt_local_port(기본 15100)도 여기서 고정된다 — Via sent-by 포트와
  소켓 bind 포트를 일치시켜 UE 응답이 올바른 소켓으로 들어오게 하기 위함.

  ---
  2단계: CaseGenerator — 테스트 케이스 조합 생성

  CampaignExecutor.run()이 CaseGenerator(config).generate()를 호출하면
  CaseSpec이 하나씩 yield된다.

  내부 로직:

  # 유효한 조합만 필터링
  for method in methods:          # ["INVITE"]
      for profile in profiles:    # ["ims_specific"]
          for layer in layers:    # ["wire"]
              for strategy in strategies:  # ["default"]
                  # 1. 이 layer에서 이 strategy가 지원되는가? (profile_catalog)
                  if strategy not in SUPPORTED_STRATEGIES_BY_LAYER[layer]:
                      continue
                  # 2. 이 profile이 이 layer+strategy 조합을 허용하는가?
                  if not profile_supports_strategy(profile, layer, strategy):
                      continue
                  yield CaseSpec(case_id, seed, profile, method, layer,
  strategy)

  max_cases까지 round-robin으로 반복. identity wire strategy는 round 0에만
  포함되고 이후 반복에서는 제외된다 (baseline은 한 번만).

  seed = seed_start + case_id — 같은 seed면 동일한 변이가 재현된다.

  ---

  퍼징

  # packet_file? (--packet-file 옵션)
  if packet_file_bytes:
      return _execute_packet_file_case(...)

  # MT template? (--mt-invite-template a31)
  if mt_template_text:

  # 나머지: 일반 PacketModel 경로

  ---
  4단계: Generator — SIP 패킷 원본 생성 (MT template 경로)

  _execute_mt_template_case() 내부:

  4-1. UE IP + IMPI resolve (케이스마다 fresh)
  resolved = ue_resolver.resolve(target, impi=impi)
  ue_ip = resolved.host      # kamctl → P-CSCF 로그 → xfrm 순
  impi = resolved.impi       # 
  "001010000123511@ims.mnc001.mcc001.3gppnetwork.org"

  4-2. port_pc / port_ps live 조회 (캐시 없음, 케이스마다)
  port_pc, port_ps = resolve_protected_ports(msisdn, ue_ip=ue_ip)
  # Strategy 1: docker logs pcscf에서 "Port is XXXXX" 파싱
  # Strategy 2: docker exec pcscf ip xfrm state → UE IP로 필터 → dport 추출
  # ue_ip를 넘기는 이유: IP가 바뀐 새 세대 포트와 이전 세대 IP가 섞이는 것 방지

  4-3. 슬롯 치환 + 렌더링
  slots = build_default_slots(
      msisdn="111111",
      impi="001010...",
      pcscf_ip="172.22.0.21",
      port_pc=8100,
      port_ps=8101,
      seed=spec.seed,          # Call-ID 등 랜덤 필드의 재현성 보장
      ...
  )
  wire_text = render_mt_invoke(template_text, slots)
  # template: mt_invite_a31.sip.tmpl (18개 {{슬롯}} 치환)
  # 마지막에 Content-Length 자동 계산 + CRLF 정규화

  4-4. 단편화 가드 체크
  if ipsec_mode == "null" and len(wire_text.encode()) > 65535:
      return verdict="unknown", reason="fragmentation-guard"

  4-5. EditableSIPMessage 파싱
  editable = parse_editable_from_wire(wire_text)
  # wire_text → header별 분리된 편집 가능한 객체로 변환
  # wire layer mutator가 이 구조를 직접 조작

  ---
  5단계: Mutator — 변이 적용

  mutated_wire = mutator.mutate_editable(editable, MutationConfig(
      seed=spec.seed,
      profile="ims_specific",    # 어떤 변이 계열을 허용할지
      strategy="default",        # "default"는 요청값 — 내부에서 concrete 
  strategy로 해석
      layer="wire",
      max_operations=1,          # --mutations-per-case
  ))

  내부에서 일어나는 일:

  strategy 결정 (resolve_effective_strategy()):
  "default" + profile="ims_specific" + layer="wire" + seed=42
  → PROFILE_DEFAULT_STRATEGY_POOLS["ims_specific"]["wire"]에서 seed로 하나 선택
  → 예: "alias_port_desync"

  layer별 변이 동작:

  - wire layer: EditableSIPMessage 구조를 텍스트 레벨로 조작
    - header_whitespace_noise: Via, Contact 등 헤더 주변 공백/탭 삽입
    - final_crlf_loss: 마지막 \r\n 삭제
    - duplicate_content_length_conflict: Content-Length 헤더를 중복 삽입 (충돌
  값)
    - alias_port_desync: Via 헤더의 ;alias= 파라미터 포트 값 비틀기
    - sdp_boundary_only / sdp_struct_only / sdp_byte_edit: SDP body 내부만 조작
  (SDP 없으면 자동 skip)
  - byte layer: 변이된 wire_text를 bytes로 변환 후 바이트 직접 조작
    - tail_chop_1: 마지막 1바이트 제거
    - tail_garbage: suffix 추가
    - header_targeted: 특정 헤더 바이트 오염

  identity strategy면 변이 없이 원본 그대로 통과.

  결과: MutatedWireCase (wire_text 또는 packet_bytes + 변이 기록 records)

  ---
  6단계: Sender — 실기기로 UDP 전송

  SendArtifact를 만들고 sender.send_artifact(artifact, mt_target)을 호출한다.

  ipsec_mode=null 경로 (_send_with_spoofed_source):

  # 1. 소스 IP를 P-CSCF IP로 bind (net.ipv4.ip_nonlocal_bind=1 필요)
  sock = socket.socket(AF_INET, SOCK_DGRAM)
  sock.bind(("172.22.0.21", 15100))   # source_ip=172.22.0.21, bind_port=15100

  # 2. Via/Contact 헤더 rewrite (--preserve-via가 없을 때)
  payload, events = prepare_real_ue_direct_payload(
      artifact,
      local_host="172.22.0.21",
      local_port=15100,
      rewrite_via=True,    # Via: SIP/2.0/UDP 172.22.0.21:15100
      rewrite_contact=True,
  )

  # 3. 전송 (UE IP:port_pc)
  sock.sendto(payload, ("10.20.20.x", 8100))

  # 4. 응답 수집 (INVITE는 collect_all=True → 1xx + 2xx 모두)
  while len(observations) < 8:
      data, addr = sock.recvfrom(65535)
      obs = parse_sip_response(data, addr)
      observations.append(obs)
      if not collect_all and obs.classification != "provisional":
          break

  라우팅 체크 (check_route_to_target):
  # UE IP까지 route가 없으면 자동으로 추가
  ip route add 10.20.20.0/24 via <gateway>

  INVITE 전송 후 CANCEL teardown (자동):
  # 180 Ringing 수신 확인 후
  cancel_text = wire_text.replace("INVITE ", "CANCEL ")
  # CSeq도 CANCEL로 변경
  sender.send_artifact(cancel_artifact, mt_target)
  # 200 OK / 487 Request Terminated 받으면 성공
  # 실패 시 3회 retry + 2초 extra cooldown

  ---
  7단계: Oracle — 응답 판정

  OracleEngine.evaluate(send_result, context)가 여러 oracle을 순서대로 확인한다:

  판정 우선순위:

  1. LogOracle (log_path 지정 시)
     → SIGSEGV, SIGABRT, Segmentation fault 등 패턴 감지
     → 즉시 "stack_failure" 반환

  2. AdbOracle (--adb 플래그 시)
     → AdbLogCollector가 스트리밍 중인 logcat 라인을 AdbAnomalyDetector에 feed
     → critical 이벤트 감지 시 → "stack_failure"
     → log_grace_seconds 동안 폴링 (SIP 응답 후 수 초 뒤에 나타나는 크래시 캐치)

  3. IosOracle (--ios 플래그 시)
     → IosSyslogCollector.slice(since, until)로 구간 syslog 추출
     → IosAnomalyDetector feed → critical 감지 시 → "stack_failure"

  4. SocketOracle (항상 실행)
     → send_result.outcome == "timeout"     → "timeout"
     → send_result.outcome == "send_error"  → "unknown"
     → response가 invalid SIP              → "suspicious"
     → 정상 응답                           → "normal"

  5. ProcessOracle (--check-process 시, 10케이스마다 or timeout 시)
     → docker inspect <container> 또는 pgrep -x <process>
     → 프로세스 없으면 → "crash"

  SocketOracle의 normal 판정 세부:
  # final_response = responses에서 마지막 non-provisional
  # 180 Ringing → provisional (수집만)
  # 200 OK → success → "normal"

  native 모드 (_send_via_native_ipsec())

  ipsec_native.py — P-CSCF netns 안에서 실제 IPsec SA를 타는 정식 경로.

  핵심 배경

  ▎ Earlier iterations used SOCK_RAW + IP_HDRINCL to spoof the protected source 
  ▎ port. That path silently bypasses Linux xfrm output processing, so "native 
  ▎ IPsec" was in fact shipping plaintext UDP — it only appeared to work against
  ▎  lenient UEs.
  ▎
  ▎ The correct approach is to use a normal kernel UDP socket inside the P-CSCF 
  ▎ netns with an explicit bind to the protected source port. The installed xfrm
  ▎  OUT policy then auto-encapsulates outbound traffic as ESP.

  즉, SOCK_RAW로 raw packet을 직접 만들면 xfrm이 무시되고 평문으로 나간다.
  반드시 SOCK_DGRAM으로 bind해야 커널의 xfrm OUT policy가 자동으로 ESP로 감싼다.

  ---
  1. IMS에서 IPsec이 왜 필요한가

  UE가 IMS에 등록(REGISTER)할 때, UE와 P-CSCF 사이에 IPsec 터널이 협상된다. 이후
   모든 SIP 트래픽은 이 터널로만 통신한다.

  UE ←─── ESP (암호화된 UDP) ───→ P-CSCF ←─── 평문 SIP ───→ S-CSCF

  UE는 자신의 IPsec SA가 설치된 포트(port_pc, port_ps)로 오는 ESP 패킷만
  받아들인다. 평문 UDP가 오면 드롭한다. 이게 bypass 모드가 spec-strict UE에서
  실패하는 이유다.

  ---
  2. Linux xfrm 프레임워크 개요

  Linux 커널에는 xfrm (transform) 이라는 IPsec 처리 서브시스템이 있다.

  두 가지 개념이 핵심이다:

  SA (Security Association) — 실제 암호화 파라미터
  src=10.20.20.2  dst=172.22.0.21
  proto=esp  spi=0xABCD1234
  알고리즘: aes-128-cbc (암호화) + hmac-sha1 (무결성)
  sport=63193  dport=5100    ← 이 포트 쌍에 대해서만 적용

  SP (Security Policy) — "이 트래픽을 어떤 SA로 처리할지" 규칙
  src=172.22.0.21  dst=10.20.20.2
  sport=5100  dport=63193
  → use SA(spi=0xABCD1234, proto=esp)  [OUT policy: 나가는 패킷]
  → use SA(spi=0xDCBA5678, proto=esp)  [IN policy: 들어오는 패킷]

  UE가 REGISTER할 때 Security-Client 헤더로 자신의 포트를 알려주고, P-CSCF가
  협상 완료 후 커널에 SA/SP를 설치한다.

  ---
  3. xfrm state 실제 예시

  docker exec pcscf ip xfrm state

  출력 예:
  src 10.20.20.2 dst 172.22.0.21    # UE → P-CSCF 방향 SA
          proto esp spi 0x0a1b2c3d
          auth hmac(sha1) ...
          enc aes ...
          sel src 10.20.20.2/32 dst 172.22.0.21/32 sport 63193 dport 5100
          #     ↑ UE port_pc                                    ↑ P-CSCF client
  port

  src 10.20.20.2 dst 172.22.0.21    # UE → P-CSCF 방향 SA (server pair)
          proto esp spi 0x0a1b2c3e
          sel src 10.20.20.2/32 dst 172.22.0.21/32 sport 61008 dport 6100
          #     ↑ UE port_ps                                    ↑ P-CSCF server
  port

  src 172.22.0.21 dst 10.20.20.2    # P-CSCF → UE 방향 SA
          proto esp spi 0x0b2c3d4e
          sel src 172.22.0.21/32 dst 10.20.20.2/32 sport 5100 dport 63193
          #     ↑ P-CSCF client port                           ↑ UE port_pc

  src 172.22.0.21 dst 10.20.20.2    # P-CSCF → UE 방향 SA (server pair)
          proto esp spi 0x0b2c3d4f
          sel src 172.22.0.21/32 dst 10.20.20.2/32 sport 6100 dport 61008
          #     ↑ P-CSCF server port                           ↑ UE port_ps

  IMS는 항상 4개 SA를 만든다. 각 방향(UE→PCSCF, PCSCF→UE) × 각 포트 쌍(client,
  server) 조합이다.

  ---
  4. MT-INVITE 전송 시 어떤 SA를 써야 하는가

  퍼저가 UE에게 MT-INVITE를 보내는 건 P-CSCF → UE 방향이다.

  P-CSCF(172.22.0.21) → UE(10.20.20.2)

  P-CSCF가 UE에게 SIP을 보낼 때는 UE의 server port(port_ps) 로 보내야 한다. UE는
   port_ps에 SIP 수신 listener를 열어두고 있다.

  따라서 써야 할 SA:
  src=172.22.0.21  sport=6100 (P-CSCF server port)
  dst=10.20.20.2   dport=61008 (UE port_ps)

  이 SA의 selector에 매칭되는 소켓이 send()를 호출해야 xfrm OUT policy가
  자동으로 ESP로 encapsulate한다.

  7. send_via_native_ipsec() — 암호화의 실제 발생 지점

  드라이버 스크립트가 P-CSCF 컨테이너 안(= P-CSCF의 network namespace)에서
  실행된다.

  # P-CSCF netns 안에서 실행되는 코드
  sock = socket.socket(AF_INET, SOCK_DGRAM)   # ← 일반 UDP 소켓

  sock.setsockopt(SO_REUSEADDR, 1)
  sock.setsockopt(SO_REUSEPORT, 1)

  sock.bind(("172.22.0.21", 6100))    # ← P-CSCF server port로 바인드
  #                          ^^^^
  #   이 포트가 xfrm OUT policy의 sport=6100과 일치해야 함

  sock.connect(("10.20.20.2", 61008)) # ← UE port_ps로 connect

  sock.send(payload)                  # ← 이 시점에 커널이 암호화

  sock.send(payload) 호출 순간 커널에서 일어나는 일:

  애플리케이션: send(payload)
        ↓
  UDP 레이어: UDP 헤더 추가
    src=172.22.0.21:6100  dst=10.20.20.2:61008
        ↓
  xfrm OUT policy 체크:
    "src=172.22.0.21 dst=10.20.20.2 sport=6100 dport=61008"
    → 매칭되는 SA 발견: spi=0x0b2c3d4f
        ↓
  ESP encapsulation:
    [IP header] [ESP header(spi)] [암호화된 UDP+SIP] [ESP trailer] [HMAC]
        ↓
  wire로 전송

  bind()한 포트가 SA selector의 sport와 일치하지 않으면 xfrm이 매칭되지 않아
  평문 UDP로 나간다. SOCK_RAW로 직접 IP 패킷을 만들면 xfrm 레이어 자체를
  우회한다. 이것이 이전 SOCK_RAW 방식이 "native IPsec인 척했지만 사실
  평문"이었던 이유다.

  ---
  8. 이중 소켓 (alt socket) — 왜 필요한가

  IMS SA는 4개고, 커널의 xfrm output policy는 SA rekey나 세션 상태에 따라 client
   SA 쌍이나 server SA 쌍 중 어느 것을 사용할지 달라질 수 있다.

  Primary socket:
    bind(pcscf_ip, 6100)   connect(ue_ip, 61008)   ← server SA 쌍
    → 보낸다

  Alt socket:
    bind(pcscf_ip, 5100)   connect(ue_ip, 63193)   ← client SA 쌍
    → 응답 대기만 (보내지 않음)

  UE의 응답이 어느 SA 쌍으로 돌아올지 모르기 때문에 두 포트 모두에서 select()로
  대기한다:

  ready, _, _ = select_mod.select([sock_primary, sock_alt], [], [], remaining)
  if ready:
      data, peer = ready[0].recvfrom(65535)  # 먼저 도착한 쪽에서 수신

  ---
  native 모드에서 UPF는 어디에 있나

  P-CSCF 컨테이너 안에서 send()가 호출될 때:

  P-CSCF 컨테이너 netns

           ↓
    [Linux 커널 IP routing]
    10.20.20.0/24 via 172.22.0.8 (UPF)
           ↓
    UPF (172.22.0.8) 수신
           ↓
    [GTP encapsulation]
    GTP-U 헤더 씌워서 RAN(srsRAN)으로 포워딩
           ↓
    RAN → LTE 라디오 → UE

  퍼저가 UPF를 직접 다루는 코드는 없다. IP 라우팅이 알아서 UPF로 보낸다. P-CSCF
  컨테이너는 Open5GS Docker 네트워크 안에 있고, 이미 10.20.20.0/24 via UPF
  경로가 설치되어 있다.
